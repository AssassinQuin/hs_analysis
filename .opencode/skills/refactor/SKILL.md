# Refactor — 通用 4 阶段代码重构工作流

适用于任意项目的系统化重构流程。每个阶段将发现存入记忆系统，支持跨会话延续。

## 工作流概览

```
预检 → 文档解析? → 阶段 1 → 关卡 → 阶段 2 → 关卡 → 阶段 3 → 关卡 → 阶段 4 → 最终验证
(会话)   (如有文档)    (研究)          (设计)          (裁剪)          (实施)
```

### 强制规则（子 Agent 必须继承）

所有子 Agent（`@explorer`/`@fixer`/`@oracle`/`@librarian`）下发任务时，prompt 中**必须包含以下全部规则**：

| 规则 | 说明 |
|------|------|
| 记忆存储 | 每个阶段发现存入记忆，标签格式：`refactor,phase-N,<module>,<type>` |
| 测试不可跳过 | 每个变更必须有验证步骤（阶段 4 每步测试必须通过） |
| Framework-First | >500 行文件必须先骨架（类/签名/docstring/import）→ 验证导入 → 再填充 |
| 语言跟随项目 | 新代码匹配项目现有风格和约定 |
| 原子提交 | 一个重构一个提交，不混合 |
| ≤3 子 Agent | 同一时刻最多 3 个并行，扁平委派（子 Agent 不再启动子 Agent） |

模板详见 [references/sub-agent-patterns.md](references/sub-agent-patterns.md)。

## 预检：会话恢复与范围确认

**启动任何阶段之前**，检查历史工作：

```
1. memory_search(tags=["refactor"], limit=10) → 之前做了什么？
2. memory_search(query="refactor phase-4 done", tags=["refactor", "phase-4"]) → 已完成的工作
3. 检查项目中是否有重构历史文件（如 references/completed-refactorings.md）→ 避免重复发现
```

**决策**：如果之前的会话完成了阶段 N，则从阶段 N+1 继续。如果阶段 4 部分完成，检查哪些待办项未完成并从那里继续。

### 范围检测

根据用户提示自动推断范围：

```
IF 提示中包含具体模块/文件名
  → 单模块
ELSE IF 提示包含 "review/审查/audit/check/检查/看看"（无动作动词）
  → 仅审查
ELSE IF 提示指出具体问题（如 "search 里的重复代码"）
  → 单问题
ELSE
  → 全项目
```

### 范围确认检查点

检测完范围后，**向用户展示推断的范围**：

```
检测到的范围：[全项目 / 单模块: X / 仅审查 / 单问题]
是否继续？[是 / 调整]
```

这防止了用户只想"重构这个函数"而 Agent 开始全项目分析的浪费。

### 范围与阶段对应

| 范围模式 | 阶段范围 | 阶段 1 范围 | 阶段 4 | 示例 |
|----------|----------|------------|--------|------|
| **全项目** | 1→4 | 整个代码库 | 所有批准项 | "重构项目代码" |
| **单模块** | 迷你-1→4 | 仅目标模块+其依赖 | 模块范围内 | "优化 search 模块" |
| **单问题** | 2→4 | 跳过（读取相关文件） | 问题范围内 | "search 里的重复代码" |
| **仅审查** | 1→3 | 整体或指定范围 | **跳过** | "审查架构" |

## 文档解析：从项目文档提取流程

**触发条件**：预检完成后，扫描项目是否存在设计文档、开发文档、重构计划、解析文档等。

### 文档发现

```
扫描以下位置，寻找项目文档：
1. docs/ 目录 — 设计文档、架构文档、API 文档
2. 根目录 *.md — README、CONTRIBUTING、CHANGELOG、计划文档
3. 包含关键词的文件 — plan, design, refactor, spec, roadmap, architecture, 决策, 方案, 计划
4. 项目管理文件 — TODO.md, ROADMAP.md, DECISIONS.md
```

### 文档分类与处理

| 文档类型 | 典型文件 | 处理方式 |
|----------|----------|----------|
| **重构计划** | `refactoring_plan.md`, `docs/refactoring/` | 直接提取计划中的阶段和步骤，映射到当前工作流 |
| **架构决策** | `DECISIONS.md`, `docs/architecture/` | 读取已有决策，避免重新讨论已定方案 |
| **设计规范** | `design/*.md`, `docs/design/` | 理解目标架构，确保重构方向一致 |
| **API 文档** | `docs/api/`, `openapi.yaml` | 了解接口契约，重构时保持兼容性 |
| **开发指南** | `CONTRIBUTING.md`, `docs/development/` | 遵循项目开发约定（编码规范、提交流程等） |
| **解析/分析文档** | `docs/analysis/`, `docs/parsing/` | 理解数据流和解析流程，重构时保持语义正确 |

### 流程提取

当发现相关文档时：

1. **读取文档** — 使用 `@explorer` 并行读取所有相关文档
2. **提取流程** — 从文档中提取：
   - 已定义的阶段/步骤（如果有重构计划）
   - 已做出的架构决策（避免重复讨论）
   - 目标架构和设计约束
   - 数据流和解析流程（如有解析文档）
   - 编码规范和项目约定
3. **生成流程摘要** — 向用户展示提取的流程：

```
📋 从项目文档中提取到以下流程：

## 已有计划（来源：<文件名>）
- 阶段 1：<描述>
- 阶段 2：<描述>
- ...

## 架构决策（来源：<文件名>）
- 决策 A：<内容>
- 决策 B：<内容>

## 开发约定（来源：<文件名>）
- 约定 1：<内容>
- 约定 2：<内容>

是否基于此流程继续？[是 / 调整 / 忽略文档从头开始]
```

4. **用户确认后**：
   - **是** → 将提取的流程映射到 4 阶段工作流，按文档计划执行
   - **调整** → 用户修改流程，按修改后的版本执行
   - **忽略** → 跳过文档，从阶段 1 全新开始
5. **存入记忆** — 将文档流程存为 `refactor,doc-flow,<source>` 标签的记忆

### 文档驱动的阶段映射

如果项目已有重构计划，将其阶段映射到本 skill 的 4 阶段：

```
项目计划的阶段 → 本 skill 阶段
项目: "阶段 1: 分析现状"  → Skill: 阶段 1 (代码考古)
项目: "阶段 2: 设计方案"  → Skill: 阶段 2 (重构发现)
项目: "阶段 3: 评审"      → Skill: 阶段 3 (裁剪)
项目: "阶段 4: 实施"      → Skill: 阶段 4 (实施)
```

如果项目计划的粒度不同，以项目计划为准，但确保每个步骤都有对应的关卡检查。

### 无文档时：自动生成重构设计文档

当项目没有设计/开发/重构文档时，按以下 **4 步快速生成**，避免在文档上花过多时间：

#### 步骤 1：代码分析（`@explorer` ×3 并行）

```
@explorer A：文件结构 + 行数 + 模块边界 + 导入依赖
@explorer B：重复代码 + 死代码 + 过大文件 + 混合职责
@explorer C：影响范围 — 谁依赖目标模块？改动波及哪些文件？
```

用中文向用户展示：目标范围、涉及文件数/行数、3-5 个关键问题、风险等级。确认后保存 `docs/<重构名称>/现状分析.md`。

#### 步骤 2：外部调研（`@librarian` ×2 并行）

```
@librarian A：成熟方案 — exa_web_search_exa + github_search_code
@librarian B：可用依赖 — github_search_repositories + zread_search_doc
```

调研维度：成熟设计模式、可替代的库、更简洁的模式、参考实现。

**反过度工程原则**：标准库 > 第三方库；if/else > 策略模式；函数 > 类；1 文件 > 包。详见 [references/anti-patterns.md](references/anti-patterns.md)。

保存到 `docs/<重构名称>/调研报告.md`。

#### 步骤 3：设计文档 → 用户中文确认

综合前两步，撰写 `docs/<重构名称>/重构设计.md`（背景 + 目标 + 当前架构问题 + 目标架构 + 实施步骤 + 不做的事 + 验证方式）。

用中文逐项确认：背景/目标 → 问题 → 目标架构 → 步骤优先级 → 范围。确认后存入记忆 `refactor,doc-design,<name>`。

#### 步骤 4：按设计执行阶段 4

每个实施步骤 = 阶段 4 一个原子变更。完成后更新 `docs/<重构名称>/进度.md`。

```
docs/<重构名称>/
├── 现状分析.md      ← 步骤 1
├── 调研报告.md      ← 步骤 2
├── 重构设计.md      ← 步骤 3（核心）
└── 进度.md          ← 步骤 4 维护
```

## 阶段 1：代码考古

**目标**：构建项目的完整地图——存在什么、如何连接、什么是死代码。

> **I/O**：输入 = 用户范围 + 记忆中的历史上下文 → 输出 = 模块地图 + 导入图 + 重复/死代码编目（存入记忆 `phase-1`）

### 步骤

1. **获取历史上下文** — `memory_search(query="refactor phase-1", tags=["refactor", "phase-1"])`
2. **映射目录结构** — 文件树含行数统计，识别模块边界
3. **构建导入依赖图** — 追踪每个源文件的入站/出站依赖
4. **运行分析清单** — [references/analysis-checklist.md](references/analysis-checklist.md)
5. **存储发现** — 每条发现：`memory_store`，标签 `refactor,phase-1,<module>,<type>`

### 关键问题

- 项目中有哪些模块？各自的职责是什么？
- 哪些文件零入站导入（死代码）？
- 代码在哪里跨模块重复？
- 最大的文件有哪些？是否混合了多种职责？

### 委派（子 Agent）

启动并行 `@explorer`（最多 3 个），使用 [references/sub-agent-patterns.md](references/sub-agent-patterns.md) 中的模板：
- Agent A：模块地图（导入、类、函数、行数统计）
- Agent B：模式搜索（重复代码检测）
- Agent C：死代码检测（零导入文件）

> **注意**：每个子 Agent 的 prompt 中必须包含「子 Agent 继承规范」中的所有规则。

### 关卡：阶段 1 完成条件

- [ ] 所有目标模块已映射并统计行数
- [ ] 导入图已构建（至少覆盖目标范围）
- [ ] 重复代码已编目
- [ ] 发现已存入记忆（带 `phase-1` 标签）

## 阶段 2：重构发现

**目标**：确定应该改变什么——消除重复、更好的模式、缺失的抽象。

> **I/O**：输入 = 阶段 1 的记忆发现 → 输出 = 重构提案列表（每条含目标文件/模式/收益/风险/工作量，存入记忆 `phase-2`）

### 步骤

1. **获取阶段 1 发现** — `memory_search(tags=["refactor", "phase-1"])`
2. **逐条分析**，寻找重构机会：
   - **重复代码** → 能否提取共享代码？
   - **缺失抽象** → Protocol/Registry 是否能简化分发？
   - **重复造轮子** → 是否已有现成的库？
   - **上帝对象** → 能否拆分职责？
3. **调研解决方案** — 使用 MCP 工具（详见 [references/mcp-tools.md](references/mcp-tools.md)）：
   - `exa_web_search_exa` 搜索成熟模式
   - `github_search_code` 搜索参考实现
   - `zread_search_doc` 查库文档指南
4. **撰写提案** — 每条包含：目标文件、模式、收益、风险、工作量
5. **存储提案** — `memory_store`，标签 `refactor,phase-2,<module>,<category>`

### 提案分类

| 分类 | 说明 |
|------|------|
| `duplication` | 相同逻辑多次实现 |
| `pattern` | 有更好的设计模式可用 |
| `dead-code` | 未使用的文件、函数或模块 |
| `missing-abstraction` | 需要统一接口的特性 |
| `test-gap` | 关键代码缺少测试覆盖 |

### 委派（子 Agent）

- `@oracle`：架构建议
- `@librarian`：库文档调研（涉及不熟悉的库时）
- 可并行——oracle 读代码，librarian 读文档

### 关卡：阶段 2 完成条件

- [ ] 每条阶段 1 发现至少有一个提案
- [ ] 提案引用了具体文件和行范围
- [ ] 适用时已调研外部方案
- [ ] 所有提案已存入记忆（带 `phase-2` 标签）

### 🛑 用户检查点：阶段 2 → 3

**裁剪前，先向用户展示提案供审阅：**

```
## 重构提案（阶段 2 输出）
| # | 分类 | 目标 | 提案 | 风险 | 工作量 |
|---|------|------|------|------|--------|
| 1 | ...  | ...  | ...  | ...  | ...    |

是否进入阶段 3（裁剪与优先级排序）？[是 / 修改 / 停止]
```

- **是** → 继续阶段 3
- **修改** → 用户指定需要调整的提案，调整后重新展示
- **停止** → 结束会话，存储发现供后续恢复

## 阶段 3：裁剪与优先级排序

**目标**：从提案中去除过度工程，排序实际值得做的工作。

> **I/O**：输入 = 阶段 2 提案 → 输出 = 分类后的提案（🔴丢弃 / 🟡简化 / 🟢保留），按优先级排序（存入记忆 `phase-3`）

### 步骤

1. **获取所有提案** — `memory_search(tags=["refactor", "phase-2"])`
2. **应用反模式检测** — [references/anti-patterns.md](references/anti-patterns.md)
3. **对每个提案提出 5 个审查问题：**
   - 这能减少总代码行数吗？
   - 是否有 >2 个开发者会接触这部分？
   - 30 天内是否有第二个消费者计划？
   - 这是否让测试更容易？
   - 新贡献者能在 5 分钟内理解吗？
4. **分类每个提案：**
   - 🔴 **丢弃** — 过度工程，无可取之处
   - 🟡 **简化** — 有价值但缩减范围
   - 🟢 **保留** — 合理，进入实施
5. **存储决策** — `memory_store`，标签 `refactor,phase-3,pruning,<verdict>`

### 核心原则

**默认选择简单。** 如果在"抽象"和"直接"之间犹豫，选直接。重构应该让代码**更短更清晰**，而不是更"架构化"。

### 不委派

阶段 3 需要项目特定的上下文判断。自己完成。

### 关卡：阶段 3 完成条件

- [ ] 每个提案已分类（丢弃/简化/保留）
- [ ] "保留"提案已按优先级排序（影响 × 难度）
- [ ] 决策已存入记忆（带 `phase-3` 标签）

### 🛑 用户检查点：阶段 3 → 4

**实施前，展示批准的重构列表：**

```
## 批准的重构（阶段 3 输出）
| # | 标题 | 文件 | 预估行数 | 优先级 |
|---|------|------|----------|--------|
| 1 | ...  | ...  | ...      | 高     |

🔴 丢弃：[列表]
🟡 简化：[列表]

是否进入阶段 4（实施）？[是 / 修改 / 停止]
```

- **是** → 开始阶段 4 实施
- **修改** → 用户调整列表，重新排序并重新展示
- **停止** → 分析完成，不改动代码

## 阶段 4：实施

**目标**：逐个执行批准的重构，每步验证。

> **I/O**：输入 = 阶段 3 批准的「保留」提案 → 输出 = 完成的重构记录（测试通过 + 行数变化，存入记忆 `phase-4`）

### 步骤

1. **获取批准的提案** — `memory_search(tags=["refactor", "phase-3"], query="keep")`
2. **按优先级排序**：高影响 + 低风险优先
3. **预检验证：**
   ```bash
   # 根据项目语言选择合适的测试命令
   # Python
   pytest tests/ -x -q
   # Node.js
   npm test
   # Go
   go test ./...
   # 通用语法检查
   python -m py_compile <changed_file>   # Python
   npx tsc --noEmit                      # TypeScript
   ```
4. **逐个执行重构：**
   a. 创建待办项
   b. **Framework-First**：文件 >500 行时，先写骨架 → 验证导入 → 填充内容
   c. 实施变更
   d. 运行测试
   e. 测试通过 → 通过 `memory_store` 记录
   f. 测试失败 → **执行回滚协议**（见下文）
5. **最终验证**：完整测试套件 + 导入检查
6. **存储完成记录** — `memory_store`，标签 `refactor,phase-4,done,<module>`

### 回滚协议

当重构导致测试失败时：

1. **立即**：`git diff` 查看变更内容
2. **评估**：是快速修复（<5 分钟）还是根本性问题？
3. **快速修复**：修复测试问题，重新运行。最多尝试 2 次。
4. **根本性问题**：回滚变更（`git checkout -- <files>`），将失败记录为发现，继续下一个重构。后续用更好的方案重试。
5. **绝对禁止**：会话结束时留下未通过的测试。

### 委派（子 Agent）

- `@fixer`：机械性多文件变更（使用 [references/sub-agent-patterns.md](references/sub-agent-patterns.md) 模板）
- `@oracle`：复杂变更后的架构审查
- 按文件夹拆分工作，实现并行 `@fixer` 实例

> **注意**：向 `@fixer` 下发任务时，prompt 中必须包含完整的目标文件列表、具体变更指令、Framework-First 规则和测试要求。

### 实施规则

- **一个重构一个提交** — 原子变更
- **每步变更后测试必须通过** — 不允许"之后修"
- **新代码遵循现有风格** — 匹配项目约定
- **最多 3 个子 Agent** — 避免上下文混乱

## 异常与降级方案

### 记忆系统不可用

当 `memory_store` / `memory_search` 失败或返回空时：

1. **降级为文件笔记** — 将发现写入 `<skill_dir>/references/session-notes.md`（不存在则创建）
2. **保留标签格式** — 使用 Markdown 标题作为标签：`## phase-1 search duplication`
3. **恢复仍可用** — 预检读取 `session-notes.md` 作为记忆的替代
4. **永不阻塞** — 如果记忆和文件都失败，继续执行但不持久化，同时警告用户

### 子 Agent 超时或失败

当 `@explorer`、`@oracle`、`@fixer` 失败/超时时：

1. **单个失败** — 缩小范围重试一次。仍失败则自己做（缩小范围可接受）
2. **全部并行失败** — 降级为串行自行执行。阶段完成时间增加但进度不断
3. **部分结果** — 使用成功 Agent 返回的任何发现。在关卡检查表中标注不完整覆盖
4. **每个 Agent 最多重试 1 次** — 不要卡在重试上。记录缺口，继续前进

### 参考文件缺失或损坏

当 `references/` 中的文件不存在时：

1. **`analysis-checklist.md` 缺失** → 阶段 1 使用 SKILL.md 中「关键问题」列表
2. **`anti-patterns.md` 缺失** → 阶段 3 使用 5 个审查问题（已在 SKILL.md 中）
3. **`completed-refactorings.md` 缺失** → 跳过历史检查，从头开始。警告用户可能重复发现
4. **`sub-agent-patterns.md` 缺失** → 使用委派策略表作为简化指引
5. **`memory-conventions.md` 缺失** → 使用各阶段步骤中展示的标签格式
6. **通用原则** — 永远不要因为参考文件缺失而中止阶段。SKILL.md 本身包含足够信息继续执行。

### 项目无测试

当项目没有测试框架或测试文件时：

1. **阶段 4 验证降级：**
   ```bash
   # Python
   python -m py_compile <changed_file>       # 语法检查
   python -c "from module import Class"      # 导入检查
   # Node.js
   node -c <changed_file>                    # 语法检查
   npx tsc --noEmit                          # 类型检查
   # Go
   go build ./...                            # 编译检查
   go vet ./...                              # 静态分析
   ```
2. **手动验证** — 逐个变更文件追踪导入链确认无断裂
3. **建议** — 会话结束时建议用户为重构模块添加测试覆盖
4. **永不跳过验证** — 即使没有测试框架，某种形式的验证是强制的

## 跨阶段模式

### 委派策略

| 阶段 | Agent | 时机 | 模板 |
|------|-------|------|------|
| 阶段 1 | `@explorer` | 并行代码库搜索 | 模块地图、模式搜索 |
| 阶段 2 | `@oracle` | 架构建议 | 架构审查、设计决策 |
| 阶段 2 | `@librarian` | 库文档调研 | （标准调研） |
| 阶段 3 | 自己 | 裁剪需要项目上下文 | — |
| 阶段 4 | `@fixer` | 机械性多文件编辑 | 多文件编辑、提取到新文件 |
| 阶段 4 | `@oracle` | 复杂变更后审查 | 架构审查 |

参见上方**预检：会话恢复与范围确认**。相同步骤：搜索记忆 → 检查阶段标签 → 读取重构历史 → 从最后完成的关卡恢复。
