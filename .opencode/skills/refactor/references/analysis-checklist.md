# 阶段 1：代码考古清单

## 目录结构审计

- [ ] 映射完整目录树，含每个文件的行数统计
- [ ] 识别顶层模块及其职责
- [ ] 标记空目录（已删除迁移的残留）
- [ ] 检查重复的目录层级（如 `engine/` vs `engine_v2/`）

## 依赖关系图

对每个源文件（根据项目语言：`.py` / `.ts` / `.go` / `.java` 等）：
- [ ] 列出所有外部导入/依赖（跨模块）
- [ ] 列出所有内部导入/依赖（同模块内）
- [ ] 统计入站依赖数（谁依赖此文件？）
- [ ] 标记**零依赖文件**（死代码候选）
- [ ] 标记**循环依赖**（A→B→A）

## 重复代码检测

- [ ] 查找跨文件的重复正则模式
- [ ] 查找功能相同但签名相似的函数
- [ ] 查找重复的数据结构定义（重叠字段）
- [ ] 查找复制的代码块（>10 行完全相同）
- [ ] 检查并行实现（OO 包装器 vs 函数式实现）

## 架构异味

- [ ] 上帝对象（文件 >800 行且混合多种职责）
- [ ] 散弹式手术（一个概念散布在 5+ 个文件中）
- [ ] 特性嫉妒（模块 A 过度访问模块 B 的内部）
- [ ] 死抽象（Protocol/接口有 0 或 1 个实现）
- [ ] 过早抽象（为单一用例构建的通用框架）

## 测试覆盖缺口

- [ ] 列出零测试覆盖的文件
- [ ] 列出只有冒烟测试（仅导入测试）的文件
- [ ] 识别缺少集成测试的关键路径

## 外部工具使用

- [ ] 列出项目中使用的所有工具/库
- [ ] 识别可以用工具/库替代的手动流程
- [ ] 检查重复造轮子（自定义实现 vs 现有库）

## 记忆存储

使用以下格式存储发现：
```
memory_store(
    content="<结构化发现>",
    metadata={
        "tags": "refactor,phase-1,<模块名>,<发现类型>",
        "type": "observation"
    }
)
```

发现类型：`dead-code`（死代码）、`duplication`（重复）、`architecture-smell`（架构异味）、`test-gap`（测试缺口）、`missing-abstraction`（缺失抽象）、`reinvented-wheel`（重复造轮子）
