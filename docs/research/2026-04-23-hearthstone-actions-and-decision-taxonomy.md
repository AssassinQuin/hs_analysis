# 炉石传说动作与决策抽象调研（含概率模型）

## 1. 调研目标

本调研面向三个目标：

1. 把炉石传说中的“所有可执行动作”抽象成统一动作体系。
2. 把“如何在当前回合规划并为后续回合做准备”抽象成决策层级。
3. 把“发现牌概率、抽牌类型概率（解场/回血/战场）”沉淀为可落地模型。

---

## 2. 基本术语与边界

为避免概念混淆，先区分四个层次：

| 层次 | 定义 | 例子 |
|---|---|---|
| 操作动作（Operation） | 玩家在客户端可直接点击的原子动作 | 打出卡牌、攻击、结束回合、发现三选一 |
| 引擎动作（Engine Action） | AI 内部统一表示的动作对象 | `Action(action_type="PLAY_WITH_TARGET", ...)` |
| 决策（Decision） | 在一个状态下对动作序列的选择 | “先解场再下怪，最后英雄技能补刀” |
| 规划（Planning） | 跨步骤/跨回合的策略组织 | “本回合保命 + 为下回合斩杀留资源” |

---

## 3. 炉石动作全量抽象（玩家视角）

## 3.1 回合内主动作

1. 打出卡牌（Play Card）
2. 使用英雄技能（Hero Power）
3. 随从攻击（Minion Attack）
4. 英雄攻击（Hero Attack，需武器或可攻击英雄形态）
5. 激活地点（Location Activate）
6. 选择发现结果（Discover Pick）
7. 选择抉择分支（Choose One Pick）
8. 结束回合（End Turn）

## 3.2 打出卡牌的子动作

打牌动作应继续细分：

1. 无目标打出（如普通随从、武器、部分法术）
2. 有目标打出（点杀法术、指向型战吼）
3. 位置相关打出（随从站位）
4. 英雄牌替换（Hero Replace）
5. 变形类施放（Transform）
6. 费用改写型打出（连锁减费、临时法力）

## 3.3 攻击动作的子动作

1. 来源维度：英雄 / 普通随从 / 风怒随从 / 突袭随从
2. 目标维度：敌方英雄 / 敌方随从（受嘲讽约束）
3. 战斗结果维度：换子 / 打脸 / 破圣盾 / 解关键威胁 / 触发亡语

## 3.4 信息动作（隐式但关键）

这些不是“点击动作”，但在决策中必须被建模：

1. 奥秘试探动作（先用低价值动作试错）
2. 资源探测动作（先看随机结果再决定后续）
3. 风险探测动作（先小攻击判断反制）

## 3.5 跨回合准备动作（战略动作）

1. 保留资源（不交关键解牌）
2. 压制资源（逼对手交解）
3. 铺场节奏控制（避免过度展开吃AOE）
4. 费用曲线预留（为下回合关键节奏点蓄力）
5. 条件机制预热（连击/延系/灌注/任务进度）

---

## 4. 决策层级抽象（从“这一步”到“后两回合”）

建议统一为 4 层决策框架：

| 层级 | 名称 | 输出 |
|---|---|---|
| L0 | 合法动作层 | 当前状态可执行的所有动作集合 |
| L1 | 回合目标层 | 本回合目标：斩杀 / 防守 / 发展 / 资源置换 |
| L2 | 动作序列层 | 在目标约束下选择最优动作序列 |
| L3 | 跨回合准备层 | 下回合计划：留牌、留费、留场、留解 |

## 4.1 回合目标（Turn Objective）定义

1. 斩杀目标（LETHAL）：优先击杀对手
2. 防守目标（DEFENSIVE）：优先保命、降低对手伤害上限
3. 发展目标（DEVELOPMENT）：最大化场面和资源效率
4. 控场目标（CONTROL）：优先解除关键威胁并建立可持续优势

## 4.2 序列规划原则（Action Sequencing）

1. 信息先行：先执行能揭示信息的动作
2. 降风险先行：先排除高风险分支（奥秘/反制/随机）
3. 费用先紧后松：先打不可替代的高费用关键动作
4. 攻击后置校验：在打牌后重新计算最优攻击顺序
5. 结束前检查：致死线、返场风险、过度展开风险

## 4.3 跨回合准备维度

1. 生存准备：本回合结束后不被反杀
2. 资源准备：下回合至少有一条高质量曲线
3. 压力准备：场面/手牌中有持续施压点
4. 反制准备：保留针对性解牌应对对手爆发回合

## 5. 概率决策抽象（发现牌与抽牌）

## 5.1 发现牌概率：按“功能类别”建模

发现决策不应仅看单卡分数，应看“命中所需功能”的概率：

功能类别建议标准化为：

1. 解场（Single/AOE Removal）
2. 回血/护甲（Heal/Armor）
3. 优质战场（High Tempo Board）
4. 资源补充（Draw/Generation）
5. 斩杀补充（Direct Damage / Burst）

定义发现池 `D`，每张候选卡 `c` 的类别集合为 `tags(c)`。

目标类别 `T` 的命中率：

`P_discover(T) = |{c ∈ D : T ∈ tags(c)}| / |D|`

若发现为“三选一”，并假设等概率采样（不放回）：

`P_hit_at_least_one(T) = 1 - C(N-K, 3) / C(N, 3)`

其中：

- `N = |D|`（池大小）
- `K =` 属于目标类别 `T` 的牌数量

## 5.2 发现动作的期望值

对目标类别设置权重 `w_T`（由当前局势决定），卡牌价值定义为：

`V(c) = Σ_T [ w_T * I(T ∈ tags(c)) ] + base_score(c)`

发现动作 EV：

`EV_discover = E[max(V(c1), V(c2), V(c3))]`

这能把“当前最需要解场还是回血”直接映射到发现选择。

## 5.3 抽牌类型概率（Topdeck Outs）

设牌库剩余 `M` 张，目标类别 `T` 有 `K_T` 张。

单抽命中率：

`P_draw_1(T) = K_T / M`

未来 `n` 抽至少命中一次：

`P_draw_n(T) = 1 - C(M-K_T, n) / C(M, n)`

多类别并集命中（如“解场或回血”）：

`P_draw_n(A ∪ B) = 1 - C(M-K_A-K_B+K_AB, n) / C(M, n)`

## 5.4 抽牌期望值

定义类型权重后，牌库平均价值：

`EV_draw_1 = (1/M) * Σ_{c∈deck} V(c)`

`EV_draw_n ≈ n * EV_draw_1`（小样本近似，可用蒙特卡洛替代）

## 6. “本回合怎么规划”可执行模板

每回合决策建议按固定步骤执行：

1. 目标判定：斩杀 > 保命 > 发展
2. 风险评估：AOE 风险、反杀风险、奥秘风险
3. 信息动作：先做能揭示信息的低成本动作
4. 主动作序列：打牌/技能/攻击的最优顺序
5. 收尾检查：是否过度展开、是否留出下回合关键资源
6. 结束回合

可导出为统一结构：

```text
TurnPlan {
  objective: LETHAL | DEFENSIVE | DEVELOPMENT | CONTROL
  primary_line: [Action...]
  backup_lines: [ActionSequence...]
  reserve_resources: [Card/HP/Mana]
  next_turn_outs: {
    clear_prob: float,
    heal_prob: float,
    board_prob: float
  }
}
```

## 7. 与当前项目的映射分析

## 7.1 已覆盖能力（现状）

| 方向 | 当前实现 |
|---|---|
| 原子动作抽象 | `Action(action_type=...)` 已统一 |
| 合法动作枚举 | `enumeration.py` 覆盖打牌/攻击/技能/地点/结束回合 |
| 序列模拟 | `simulation.py` 覆盖主要动作状态转移 |
| 回合目标 | `strategic.py` 已有 LETHAL/DEFENSIVE/DEVELOPMENT |
| 攻击规划 | `attack_planner.py` 提供贪心攻击序列 |
| 动作剪枝 | `action_pruner.py` 做支配与冗余过滤 |
| 风险评估 | `risk_assessor.py` 覆盖 AOE/过铺/生存/奥秘风险 |
| 发现EV | `discover_model.py` 提供 discover EV 与采样评估 |
| 抽牌EV | `draw_model.py` 提供抽牌期望和阈值命中概率 |

## 7.2 关键缺口（面向“完整动作/决策”）

1. 动作类型仍是字符串，缺少强类型枚举（易拼写漂移）。
2. “功能类别标签（解场/回血/战场）”尚未系统化沉淀到所有卡牌。
3. 发现概率与抽牌概率未统一到同一个“需求感知权重模型”。
4. 跨回合规划尚以启发式为主，缺少显式 `TurnPlan` 输出对象。
5. 隐藏信息动作（奥秘试探、读手）尚未作为独立动作层建模。

## 8. 推荐的统一动作字典（建议落地）

建议将动作抽象统一为以下最小闭包：

1. `PLAY_CARD`
2. `PLAY_CARD_TARGETED`
3. `ATTACK_MINION`
4. `ATTACK_HERO`
5. `USE_HERO_POWER`
6. `ACTIVATE_LOCATION`
7. `HERO_REPLACE`
8. `DISCOVER_PICK`
9. `CHOOSE_ONE_PICK`
10. `END_TURN`
11. `INFO_PROBE`（信息探测动作）
12. `RESOURCE_HOLD`（保留资源动作）

其中 11/12 可先作为“元动作标签”，不改底层执行器，只用于上层规划解释。

## 9. 概率面板建议（用于日志与UI）

每个我方回合输出以下指标，便于观察“计划是否合理”：

1. `P_clear_1`：下回合单抽解场概率
2. `P_heal_1`：下回合单抽回血概率
3. `P_board_1`：下回合单抽优质战场概率
4. `P_clear_2`：未来两抽至少命中解场概率
5. `P_lethal_out`：对手下回合斩杀概率估计
6. `discover_hit_clear`：当前发现池命中解场概率
7. `discover_hit_heal`：当前发现池命中回血概率

## 10. 结论

本项目已经具备“动作枚举 + 序列模拟 + 风险评估 + 概率子模块”的核心骨架。

下一步的关键不是继续堆动作数量，而是把三件事统一：

1. 用统一动作字典标准化执行层与解释层；
2. 用统一“功能类别标签”贯通发现概率、抽牌概率、回合目标；
3. 用显式 `TurnPlan` 输出把“当前动作”与“后续准备”连接起来。

完成后，系统可从“会算当前最优步”升级为“可解释的中期规划引擎”。

## 11. 外部深度调研补充（非仅项目内）

本节基于 HearthSim 协议文档、官方站点、社区规则文档做交叉整理，作为动作抽象的外部依据。

## 11.1 玩家动作与输入循环

外部资料对“玩家可执行动作”与“服务器判定动作”的共识如下：

1. 玩家回合是服务器给出可选项（Options）后进行输入，核心动作包含：打牌、攻击、使用英雄技能、结束回合。
2. 实际可用动作由服务器下发的 options 决定，而不是客户端本地任意执行。
3. 结束回合（END_TURN）是特殊选项，通常始终存在于可选项列表中。

这与项目中 `enumerate_legal_actions(state)` 的设计一致（先枚举合法，再搜索）。

## 11.2 协议层动作与状态变更

HearthSim 的 Game State Protocol 给出了动作在日志中的状态表达：

1. 状态包核心类型：`FULL_ENTITY`、`SHOW_ENTITY`、`HIDE_ENTITY`、`TAG_CHANGE`、`BLOCK_START`、`BLOCK_END`、`CREATE_GAME`、`META_DATA`。
2. 所有动作最终都可映射为“实体标签变化 + 区域迁移（ZONE transition）”。
3. 块结构（BLOCK）表达了“动作触发动作”的因果链，适合回放器做嵌套因果还原。

这直接支撑“动作抽象 = 行为语义层 + 协议状态层”的双层模型。

## 11.3 发现（Discover）机制关键规则

社区规则文档对 Discover 的关键描述可归纳为：

1. 发现是“三选一”，候选来自受限池（职业、费用、种族等过滤条件）。
2. 默认情况下选择结果加入手牌（除非卡牌文本指定了其他落点，比如直接召唤或洗入牌库）。
3. 常规 Discover 不提供重复选项（3个选项互不重复）。
4. 回合计时结束未选择时，系统会随机代选。

这说明“发现概率”必须建模为“受约束抽样 + 三选一最大化”而不是单抽。

## 11.4 攻击与结算顺序

社区高级规则和攻击条目强调：

1. 攻击是独立序列（Sequence），具有准备与战斗等阶段。
2. 伤害交换后才会进入死亡相关结算（Deathrattle/死亡触发/相关奥秘）。
3. 多个死亡触发按入场顺序等规则串行结算，顺序会影响最终结果。

这与项目中“攻击后统一死亡清理→亡语→复生→光环重算”的流程方向一致。

## 11.5 抽牌概率的外部数学方法

外部卡牌游戏研究普遍使用超几何分布（Hypergeometric Distribution）处理“在未来 n 抽内命中特定类别”的问题。

因此本文第 5 章给出的以下公式具备跨游戏通用性，可作为炉石实现基线：

1. 单抽命中：`K / M`
2. n 抽至少一次命中：`1 - C(M-K, n) / C(M, n)`
3. 多类别并集命中：`1 - C(M-K_union, n) / C(M, n)`（按交并关系修正）

## 12. 调研结论升级（结合外部资料后的落地建议）

结合外部资料后，建议把“动作抽象”正式升级为三层：

1. **玩家动作层（UI语义）**：打牌、攻击、技能、结束回合、发现选择等。
2. **策略动作层（决策语义）**：探测、保留资源、骗解、压节奏、反斩准备等。
3. **协议动作层（日志语义）**：实体创建/揭示/标签变化/块级因果链。

并且把“概率决策”升级为统一接口：

1. `P_discover(role)`：发现命中某功能类型（解场/回血/战场）的概率
2. `P_draw_n(role)`：未来 n 抽命中某功能类型概率
3. `EV_plan`：当前主线动作 + 未来 outs 的联合期望

当三层打通后，系统将不仅能输出“怎么打”，还能输出“为什么这样打、这样打对下回合概率有何影响”。
