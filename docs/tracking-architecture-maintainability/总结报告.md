# 对手追踪架构可维护性 调研总结报告

> 生成日期: 2026-04-24
> 研究范围: 新增追踪规则（洗入牌库/腐蚀/条件牌贝叶斯/累计机制）的可维护架构设计

## 一、研究背景与目标

### 问题陈述

在上一轮开发中，我们为对手追踪系统新增了4种追踪规则：

1. **洗入牌库追踪** — 检测`*→DECK`区域转换，记录对手洗入牌库的已知牌
2. **腐蚀升级追踪** — 检测`SHOW_ENTITY`中对手手牌card_id变更（Corrupt升级）
3. **条件牌贝叶斯证据** — `KnownCard`新增条件证据字段，`BayesianOpponentModel`新增`conditional_evidence()`方法
4. **累计机制桥接** — `OpponentState`新增opp_corpses/opp_herald_count/opp_quests等字段，`StateBridge`从GlobalTracker同步

每次新增追踪关注点都需要修改**至少4个文件**（GlobalGameState→OpponentState→StateBridge→GameState.copy()），涉及33+字段的手动协调。`hasattr()`防护的使用表明schema正在变得脆弱。

### 期望目标

- **新增追踪规则 ≤ 1个文件**：定义追踪规则只需一个地方
- **删除追踪规则 ≤ 1个文件**：不影响其他规则
- **修改追踪规则 ≤ 1个文件**：条件/行为的变更是局部的
- **字段桥接自动化**：新增字段不需要手动写拷贝代码
- **向后兼容**：不破坏packet_replayer路径和现有736个测试
- **性能不退化**：GameState.copy()仍满足MCTS每秒数千次调用的需求

## 二、现有项目基础

### 当前架构

追踪数据流经4层：

```
GlobalTracker (事件处理) → GlobalGameState (数据存储) → StateBridge (桥接) → GameState/OpponentState (搜索使用)
```

每层的变更点：

| 层 | 文件 | 新增追踪时需修改 |
|----|------|-----------------|
| 事件处理 | global_tracker.py `on_zone_change()`/`on_show_entity()` | 插入if/elif分支 |
| 数据存储 | global_tracker.py `GlobalGameState` | 添加字段 |
| 桥接 | state_bridge.py `_enrich_from_global_state()` | 手动拷贝字段 |
| 搜索状态 | game_state.py `OpponentState` + `GameState.copy()` | 添加字段+手动深拷贝 |

### 五大痛点

**痛点1: if/elif单体链** — `on_zone_change()`有9个分支、`on_show_entity()`有4个关注点混合，新增区域处理需要在链中插入新分支 [P1]

**痛点2: 字段爆炸** — GlobalGameState 33字段、OpponentState 13字段，每新增关注点需4+文件变更 [P1][P3][P2]

**痛点3: 双边字段重复** — 7+个player/opponent对偶字段（corpses/herald/quests/shuffled/stats/generated/cards_played），无`PerSide<T>`抽象 [P1]

**痛点4: 手动深拷贝** — `GameState.copy()`手动枚举24个可变字段，漏拷贝会导致MCTS搜索结果静默错误 [P3]

**痛点5: 双重存储** — MechanicsState和GameState原始字段存储相同数据，需同步 [P4]

### 已有可复用组件

项目中已存在3种可扩展模式，可直接借鉴：

1. **Mechanic Protocol + MechanicDispatcher** [P5][P6] — Observer/Strategy混合，7个事件方法，5个适配器已实现。**最适合追踪层复用**。

2. **EffectKind注册表 + @register装饰器** [P7] — 替代了deathrattle/trigger/location中的3×重复分发。**适合区域处理分发**。

3. **数据驱动目标解析** [spell_target_resolver.py] — 条件谓词作为可调用对象组合。**适合条件牌证据定义**。

## 三、外部研究综合

### 3.1 Fireplace事件驱动架构 [E1][E4]

Fireplace（HearthSim官方Python模拟器）的核心设计：

- **Action对象**封装所有状态变更（Play/Attack/Hit/Heal/Summon等）
- **Action执行后自动广播**给注册了对应事件的EventListener
- **Selector机制**：事件触发后用Selector过滤匹配的实体，而非在处理逻辑中写条件
- **声明式事件绑定**：卡牌定义中写`events=[events.Play(selector=...)`，而非在处理器中写if

对我们的启示：
- **当前问题对应**：我们的`on_zone_change()`是在处理器内部用if/elif判断"什么转换发生了"，而Fireplace是"转换发生了，谁关注它"
- **可借鉴**：将区域转换作为事件广播，追踪规则作为监听者注册

### 3.2 dispatch_dict模式 [E6]

python-hslog的`BaseExporter`用字典替代条件链：

```python
# hslog的方案
dispatch = {
    PacketType.FULL_ENTITY: self._export_full_entity,
    PacketType.SHOW_ENTITY: self._export_show_entity,
    PacketType.TAG_CHANGE: self._export_tag_change,
}
handler = dispatch.get(packet.type)
if handler:
    handler(packet)
```

对我们的启示：
- **直接适用**：`on_zone_change()`的9分支if/elif → `{(old_zone, new_zone): handler}`字典
- **已有先例**：项目effects.py的`_REGISTRY`已采用相同模式

### 3.3 ECS核心原则 [E7][E8]

Entity-Component-System的关键约束：

- **所有权规则**：每个组件只有一个写入者（System）
- **组件=纯数据**：不含行为逻辑
- **系统=无状态逻辑**：操作匹配的组件集合
- **Push/Pull混合**：热循环用Pull（系统查询），边界用Push（事件通知）

对我们的启示：
- **当前问题对应**：GlobalTracker和StateBridge都可能写同一字段，违反所有权规则
- **可借鉴**：追踪规则=System（逻辑），GlobalGameState的字段=Component（数据），GlobalTracker=Entity管理器
- **但需注意**：完整ECS对Python项目过于复杂，应取其精华（所有权+组合），而非照搬架构

### 3.4 观察者模式在游戏中的经典应用 [E9]

观察者模式的核心价值：
- **解耦**：被观察者不关心谁接收通知
- **动态注册/注销**：运行时添加/移除
- **已有实现**：项目MechanicDispatcher本质上就是观察者模式的变体

## 四、方案设计

### 方案A: Tracker Protocol + 区域处理注册表

**核心思想**：定义`TrackerRule`协议，每条追踪规则实现为独立的规则类，通过注册表分发事件。

```python
# analysis/watcher/tracker_rules.py

class TrackerRule(Protocol):
    """一条追踪规则的协议接口"""
    name: str  # 规则名称，用于日志和调试

    def on_zone_change(self, ctx: TrackingContext, entity_id: int,
                       controller: int, old_zone: int, new_zone: int,
                       card_id: str, card_type: int) -> None: ...

    def on_show_entity(self, ctx: TrackingContext, entity_id: int,
                       card_id: str, controller: int,
                       zone: int, card_type: int, cost: int) -> None: ...

    def on_card_played(self, ctx: TrackingContext, entity_id: int,
                       controller: int, card_id: str, card_type: int) -> None: ...


@dataclass
class TrackingContext:
    """追踪上下文，包含对GlobalGameState的引用和辅助方法"""
    state: GlobalGameState
    our_controller: int
    opp_controller: int
    entity_birth: dict
    card_metadata_fn: Callable

    @property
    def is_opp(self, controller: int) -> bool:
        return controller == self.opp_controller
```

**区域处理注册表**：
```python
# 或者用更轻量的dispatch_dict替代on_zone_change的if/elif
_ZONE_HANDLERS: Dict[Tuple[int, int], List[Callable]] = {}

def register_zone_handler(old_zone: int, new_zone: int):
    def decorator(fn):
        key = (old_zone, new_zone)
        _ZONE_HANDLERS.setdefault(key, []).append(fn)
        return fn
    return decorator
```

**示例规则实现**：
```python
# analysis/watcher/tracker_rules/shuffle_rule.py

class ShuffleIntoDeckRule:
    """洗入牌库追踪规则"""
    name = "shuffle_into_deck"

    def on_zone_change(self, ctx: TrackingContext, entity_id, controller,
                       old_zone, new_zone, card_id, card_type):
        if new_zone == ZONE_DECK and card_id:
            if ctx.is_opp(controller):
                ctx.state.opp_shuffled_into_deck.append(card_id)
            else:
                ctx.state.player_shuffled_into_deck.append(card_id)
```

**优点**：
- 新增规则 = 新文件 + 注册，不动现有代码
- 规则之间完全隔离，删除/修改不影响其他规则
- 与Mechanic Protocol风格一致，团队熟悉

**缺点**：
- 每个事件方法对所有规则都会调用（即使规则不关心），有性能开销
- TrackerRule接口需要覆盖所有可能的事件，新增事件类型时所有规则都要更新
- 字段桥接问题未解决（仍需手动拷贝GlobalGameState→OpponentState）

### 方案B: 轻量ECS — TrackingComponent + TrackingSystem

**核心思想**：将GlobalGameState拆分为独立的Component，每个System负责一种追踪关注点的读写。

```python
# Component = 纯数据
@dataclass
class ShuffleComponent:
    opp_shuffled: List[str] = field(default_factory=list)
    player_shuffled: List[str] = field(default_factory=list)

@dataclass
class CorruptComponent:
    opp_corrupted_cards: List[str] = field(default_factory=list)
    opp_corrupted_upgrades: Dict[str, str] = field(default_factory=dict)

@dataclass
class MechanicsComponent:
    player_corpses: int = 0
    opp_corpses: int = 0
    player_herald: int = 0
    opp_herald: int = 0
    ...

# System = 逻辑
class ShuffleSystem:
    COMPONENT_TYPE = ShuffleComponent

    def on_zone_change(self, component: ShuffleComponent, event):
        if event.new_zone == ZONE_DECK and event.card_id:
            if event.is_opp:
                component.opp_shuffled.append(event.card_id)
            else:
                component.player_shuffled.append(event.card_id)

# GlobalGameState变为Component容器
class GlobalGameState:
    def __init__(self):
        self._components: Dict[Type, Any] = {}

    def get_component(self, comp_type: Type[T]) -> T:
        return self._components.get(comp_type)
```

**优点**：
- 所有权清晰：每个System只写自己的Component
- 组件可组合：新增关注点 = 新Component + 新System
- 序列化/桥接可基于Component类型自动化

**缺点**：
- 架构变化过大（GlobalGameState从33字段的dataclass → Component容器）
- GameState.copy()需要遍历所有Component，性能影响需评估
- 现有736个测试全部依赖当前GlobalGameState结构，迁移成本高
- Python中ECS的性能优势（缓存局部性）无法体现

### 方案C: 渐进式重构 — 扩展现有MechanicDispatcher

**核心思想**：不引入新架构，而是将追踪层的3个具体问题逐一解决，利用已有模式。

#### C1: 区域处理 → dispatch_dict
```python
# global_tracker.py
class GlobalTracker:
    _ZONE_HANDLERS: Dict[Tuple[int, int], List[Callable]] = {}

    @classmethod
    def register_zone_handler(cls, old_zone: int, new_zone: int):
        """装饰器：注册区域转换处理器"""
        def decorator(fn):
            cls._ZONE_HANDLERS.setdefault((old_zone, new_zone), []).append(fn)
            return fn
        return decorator

    def on_zone_change(self, entity_id, controller, old_zone, new_zone, card_id, card_type):
        # 公共逻辑（opp_hand_card_ids更新）
        ...

        # 分发给注册的处理器
        handlers = self._ZONE_HANDLERS.get((old_zone, new_zone), [])
        for handler in handlers:
            handler(self, entity_id, controller, old_zone, new_zone, card_id, card_type)

        # 也检查通配处理器（new_zone匹配）
        for handler in self._ZONE_HANDLERS.get((None, new_zone), []):
            handler(self, entity_id, controller, old_zone, new_zone, card_id, card_type)
```

#### C2: 字段桥接 → 声明式映射
```python
# state_bridge.py
_OPP_FIELD_MAP: List[Tuple[str, str]] = [
    ("opp_corpses", "opp_corpses"),
    ("opp_herald_count", "opp_herald_count"),
    ("opp_shuffled_into_deck", "opp_shuffled_into_deck"),
    ("opp_corrupted_cards", "opp_corrupted_cards"),
    ("opp_weapon", "opp_weapon_card_id"),  # 字段名映射
    ("opp_secrets", "secrets"),
    ...
]

def _enrich_from_global_state(self, opp_state, game_state, global_state):
    for gs_field, opp_field in _OPP_FIELD_MAP:
        gs_value = getattr(global_state, gs_field, None)
        if gs_value is not None:
            setattr(opp_state, opp_field, gs_value)
```

#### C3: GameState.copy() → dataclass introspection
```python
# game_state.py
def copy(self) -> "GameState":
    """使用dataclass fields自动深拷贝，无需手动枚举。"""
    import dataclasses
    gs = GameState()
    for f in dataclasses.fields(self):
        val = getattr(self, f)
        if isinstance(val, list):
            setattr(gs, f.name, list(val))
        elif isinstance(val, set):
            setattr(gs, f.name, set(val))
        elif isinstance(val, dict):
            setattr(gs, f.name, dict(val))
        else:
            setattr(gs, f.name, val)  # int/str/bool/None — immutable

    # OpponentState同样处理
    opp = dataclasses.replace(self.opponent)
    for f in dataclasses.fields(opp):
        if f.name in ('hero',):
            continue  # 特殊处理
        val = getattr(self.opponent, f.name)
        if isinstance(val, list):
            setattr(opp, f.name, list(val))
        elif isinstance(val, dict):
            setattr(opp, f.name, dict(val))
    gs.opponent = opp
    gs._mechanics = self._mechanics.copy() if self._mechanics else None
    return gs
```

**优点**：
- 变更最小，可逐步实施
- 每个子问题独立解决，不需要大重构
- 向后兼容（不改变GlobalGameState结构）
- dispatch_dict已有项目先例（effects.py）

**缺点**：
- 仍然是"改良"而非"根治"，字段仍需在多处声明
- dispatch_dict对通配匹配（如"任意区域→DECK"）支持不如Protocol灵活
- dataclass introspection的copy()可能有边界情况（新增不可变类型需注意）

### 方案对比

| 维度 | A: Tracker Protocol | B: 轻量ECS | C: 渐进式重构 |
|------|-------------------|-----------|-------------|
| **新增规则文件变更** | 1个新文件 | 1个新文件 | 2-3个文件 |
| **删除规则影响** | 删1个文件 | 删1-2个文件 | 改2-3个文件 |
| **迁移成本** | 中（新建规则文件+注册） | 高（GlobalGameState重写） | 低（逐步替换） |
| **现有测试影响** | 中（需验证行为一致） | 高（736测试全受影响） | 低（可逐步验证） |
| **性能影响** | 中（每事件遍历所有规则） | 低（Component查询快） | 低（dispatch_dict O(1)查找） |
| **与现有代码风格一致性** | 高（类似Mechanic Protocol） | 低（新架构范式） | 高（扩展现有模式） |
| **根治问题程度** | 高（规则隔离+注册） | 最高（完全解耦） | 中（改良但不根治） |
| **实施风险** | 低 | 高 | 最低 |

## 五、实施建议

### 推荐: 方案A（Tracker Protocol）+ 方案C的C2/C3（声明式桥接+自动拷贝）

**理由**：
- 方案A解决核心问题（规则隔离），方案C解决工程问题（桥接自动化）
- 两者正交，可同时实施
- 避免方案B的大规模架构变更风险

### 迁移路径（5步，可独立PR）

#### Step 1: dispatch_dict替代on_zone_change的if/elif链
- **文件**: global_tracker.py
- **变更**: 将9分支if/elif提取为9个独立方法，用`_ZONE_HANDLERS`字典分发
- **影响**: 0个外部文件
- **验证**: 现有736测试通过

#### Step 2: TrackerRule协议 + TrackingContext
- **新建文件**: `analysis/watcher/tracker_rule.py`
- **变更**: 定义TrackerRule Protocol、TrackingContext dataclass
- **影响**: 0个现有文件（纯新增）

#### Step 3: 迁移洗入牌库/腐蚀规则为独立TrackerRule
- **新建文件**: `analysis/watcher/tracker_rules/shuffle_rule.py`, `corrupt_rule.py`
- **变更**: 从on_zone_change/on_show_entity中提取对应逻辑
- **影响**: global_tracker.py减少代码（逻辑移到规则文件）
- **验证**: test_opponent_tracking.py的22个测试通过

#### Step 4: 声明式字段桥接
- **文件**: state_bridge.py
- **变更**: `_OPP_FIELD_MAP`列表 + `_enrich_from_global_state`用循环替代手动拷贝
- **影响**: 0个外部文件
- **验证**: test_opponent_tracking.py通过

#### Step 5: GameState.copy()自动化
- **文件**: game_state.py
- **变更**: 用dataclass.fields()自动拷贝替代手动枚举
- **影响**: 0个外部文件
- **风险**: 需仔细测试MCTS性能（copy()是热路径）
- **验证**: 全部测试通过 + MCTS benchmark对比

### 后续追踪规则的新增流程（目标达成后）

```
1. 创建 analysis/watcher/tracker_rules/new_rule.py
2. 实现 TrackerRule 接口的 on_zone_change/on_show_entity/on_card_played
3. 在 GlobalTracker.__init__() 中注册: self._rules.append(NewRule())
4. 在 GlobalGameState 中添加必要的数据字段
5. 在 state_bridge.py 的 _OPP_FIELD_MAP 中添加映射行
6. 完成 — 不需要修改 on_zone_change/on_show_entity/if/elif链
```

从"4+文件手动协调"降低到"1个规则文件 + 1行注册 + 1行映射"。
