# 外部研究资料: 追踪架构可维护性

> 检索日期: 2026-04-24

## 来源1: Fireplace 事件驱动架构 [E1][E2]

- URL: https://deepwiki.com/jleclanche/fireplace
- 关键发现:
  - **EventListener + Action管道**: 所有游戏状态变更通过Action对象流转，Action执行后自动广播给匹配的EventListener
  - **声明式事件绑定**: 卡牌定义中用`events=`注册触发器，触发器用Selector匹配目标实体
  - **Action广播机制**: Play(卡牌) → broadcast → 所有注册了`OnPlay`的Listener收到通知 → 用Selector过滤是否匹配
  - **ActionArg延迟求值**: Action参数在执行时才解析，支持动态上下文
  - **GameAction vs TargetedAction**: 直接游戏状态修改 vs 针对特定目标的操作
- 可信度: 高 (官方HearthSim项目，代码实际运行)

## 来源2: HearthSim 游戏状态协议 [E3][E4]

- URL: https://hearthsim.info/docs/gamestate-protocol/
- 关键发现:
  - **状态包(Power Packets)**: FULL_ENTITY/SHOW_ENTITY/TAG_CHANGE 三种包类型足以描述所有状态变化
  - **不对称信息**: 服务器全局状态 vs 客户端已知状态 — dispatcher为每个玩家过滤/保留不同的包
  - **Zone层次**: PLAY/DECK/HAND/GRAVEYARD/SETASIDE/SECRET 六个区域
  - **Action与Event解耦**: Action是"做了什么"，Event是"发生了什么" — 同一Action可产生多个Event
- 可信度: 高 (HearthSim社区标准，被HDT等工具使用)

## 来源3: 卡牌游戏AI架构 (Benny Cheung) [E5]

- URL: https://bennycheung.github.io/game-architecture-card-ai-1
- 关键发现:
  - **Game Model = 信息系统**: 维护游戏宇宙的状态，玩家通过合法操作序列交互
  - **分层架构**: Game Model → Game Engine → Game Interface → Game AI，每层职责明确
  - **Card Design驱动复杂度**: 卡牌属性（type/cost/powers/flags）决定游戏逻辑复杂度
  - **Player状态**: actions/prev_actions/phase_bonus等字段 — 与我们的OpponentState类似
  - **Game全局状态**: turn/round/game_over/action_selected — 与我们的GameState类似
- 可信度: 高 (实际RFTG AI项目，有完整源码和论文)

## 来源4: ECS架构核心模式 [E7][E8]

- URL: https://pulsegeek.com/articles/ecs-architecture-in-game-development-core-patterns
- 关键发现:
  - **Entity=ID, Component=Data, System=Logic**: 严格分离，减少偶然耦合
  - **所有权规则**: 每个组件类型只有一个写入者，避免冲突
  - **Push/Pull混合**: 热循环用Pull（系统查询），边界用Push（事件通知）
  - **组合优于继承**: 新机制 = 混合已有组件，而非新子类
  - **测试性**: 系统可用mock组件数据独立测试
- 可信度: 高 (架构模式行业共识)

## 来源5: Game Programming Patterns: Observer [E9]

- URL: https://gameprogrammingpatterns.com/observer.html
- 关键发现:
  - **解耦通知与响应**: 被观察者不关心谁接收通知
  - **动态注册/注销**: 运行时添加/移除观察者
  - **经典问题**: 观察者列表的迭代安全性（注销时），观察顺序的不确定性
- 可信度: 高 (经典设计模式参考)

## 来源6: python-hslog导出器 [E6]

- URL: https://github.com/HearthSim/python-hslog
- 关键发现:
  - **BaseExporter + dispatch_dict**: 每种包类型映射到处理方法，通过字典查找分发
  - **PacketTree递归导出**: 树结构遍历，每个节点调用对应的export方法
  - **dispatch_dict模式**: 替代if/elif链的经典方案 — `{PacketType.FULL_ENTITY: self._export_full_entity}`
- 可信度: 高 (本项目已依赖hslog库)

## 综合发现

### 发现1: 事件驱动是CCG模拟器的事实标准 (E1+E2+E5)
Fireplace、SabberStone、hearthstone-lab等HS模拟器都使用事件驱动架构。核心模式：
- **Action对象**封装状态变更
- **事件广播**通知所有监听者
- **Selector/Condition**过滤匹配

### 发现2: dispatch_dict替代if/elif是低风险迁移 (E6)
python-hslog的BaseExporter用`{type: handler}`字典替代条件链。这与项目已有的effects.py注册表模式一致。

### 发现3: ECS的"所有权规则"适用 (E7+E8)
每个追踪字段应该只有一个"权威写入者"。目前GlobalTracker、StateBridge、GameState.copy()都可能写入同一字段，导致同步问题。

### 发现4: 声明式事件绑定优于过程式 (E1+E4)
Fireplace在卡牌定义中声明`events=...`而非在处理器中写if条件。这种"注册时确定关注点"的模式比"在处理链中写条件"更可维护。

### 发现5: Push/Pull混合模式适合追踪场景 (E7)
- **Push**: 实时日志事件触发追踪更新（on_zone_change/on_show_entity）
- **Pull**: 搜索引擎按需读取追踪状态（StateBridge.convert()）
两种模式各司其职，无需统一为单一模式。
