### 4. 效果执行现状

**spell_simulator.EffectApplier** — 法术效果的主要执行器：
- 从 `CardEffects` 结构化数据提取效果类型 (damage/summon/draw/heal/armor/buff/discard/cost_reduce)
- 有 `_from_text_fallback()` 用 regex 的备选方案
- 应用效果到 GameState：aoe_damage, random_damage, direct_damage, summon, draw, destroy, heal, armor, buff, discard, cost_reduce

**battlecry_dispatcher** — 战吼效果：
- 用 regex 提取战吼文本（中英文正则）
- 检查 9 种战吼子类型（destroy_minion, freeze, silence, give_divine_shield, give_taunt, give_charge, give_rush, discover, copy_minion）
- 复用 spell_simulator 的 EffectParser + EffectApplier

**deathrattle** — 亡语效果：
- 已迁移到 abilities 系统
- 先查 enchantment 触发器，再查 abilities DEATHRATTLE trigger

## 三、可复用组件

### A. 直接复用
1. **abilities/definition.py 的类型系统** — 4 枚举 + 4 数据类设计良好，可直接作为新系统的数据模型
2. **abilities/tokens.py 的映射表** — MECHANICS_TRIGGER_MAP, ACTION_VERB_MAP 等可作为扩展基础
3. **abilities/extractors.py 的字符串提取** — 纯函数，可复用
4. **abilities/executor.py 的执行框架** — trigger → check_condition → execute_effects 流程可扩展

### B. 需要整合
1. **spell_simulator.EffectParser/Applier** — 与 abilities executor 功能重叠，需要合并
2. **battlecry_dispatcher 的战吼文本提取** — 需要纳入 abilities parser
3. **effects.py 的 parse_effect/dispatch** — 独立的效果分发系统，需要统一

### C. 保持独立
1. **aura_engine** — 光环效果需要持续重算，不适合 parse-once 模型
2. **trigger_system** — 触发器注册和分发是运行时行为
3. **discover.py** — 发现池计算是数据查询，不是效果解析
4. **各机制模块 (colossal, corpse, dormant 等)** — 各有特定逻辑，不直接纳入解析器

## 四、经验与约束

### 设计约束
1. **英语卡牌文本** — abilities parser 目前只解析 `englishText`（`enUS`），中文文本仅用于 battlecry_dispatcher 的 regex 匹配
2. **零正则原则** — abilities 系统明确使用 `string.find()`/`split()`，但 spell_simulator 和 battlecry_dispatcher 仍大量使用 regex
3. **parse-once, execute-many** — CardAbility 缓存在 Card 对象上，避免重复解析
4. **不可变模拟** — apply_action 总是拷贝 GameState

### 已知限制
1. **Parser 覆盖率 60.8%** — 147 张有 mechanics 的卡未被解析
2. **Effect 执行不完整** — DISCOVER/DESTROY/FREEZE/SILENCE 是空操作
3. **两套效果系统并存** — abilities executor vs spell_simulator EffectApplier，逻辑重叠
4. **静态关键词不产生 ability** — TAUNT/RUSH/LIFESTEAL/STEALTH/DIVINE_SHIELD 等仅作为标签
5. **缺少目标解析** — 目标选择依赖 SpellTargetResolver（独立模块），abilities 的 TargetSpec 仅用于文本匹配
6. **无正式语法** — 没有 BNF/ANTLR，纯 keyword table + string.find()，难以扩展

### 已有设计决策
1. 选择 abilities 统一系统作为效果解析方向（deathrattle.py 已迁移）
2. RHEA 引擎已删除，只保留共享原语（Action/Enumeration/Simulation）
3. simulation.py 是**编排层**，不实现具体效果，委托给机制模块

## 五、知识空白

1. **如何设计通用语法** — 当前是 keyword table 匹配，需要设计能容纳所有标准卡效果的语法体系（可能是扩展的 token 匹配 + 条件树）
2. **静态关键词如何统一** — TAUNT/RUSH 等不产生 ability，但模拟中需要处理它们（如 Taunt 影响攻击目标选择）
3. **如何消除 spell_simulator 与 abilities 的重叠** — 两套效果执行系统，需要确定哪个是唯一权威
4. **复杂效果的表示** — 如 "If X then Y, otherwise Z" 的条件分支效果、嵌套效果、连锁效果
5. **目标选择的统一模型** — 当前 TargetSpec 是文本匹配，实际目标解析在 SpellTargetResolver
6. **如何处理运行时状态依赖** — "for each X" 需要数场上实体，"if your hand is empty" 需要检查手牌
7. **跨卡协同效果** — 如 AURA 影响其他卡、光环重算、触发器链
8. **发现(Discover)的模拟策略** — 发现是随机从池中选卡，模拟中如何近似？
9. **变换(Transform)效果的完整语义** — 当前只是变 1/1 Sheep，实际需要保留/移除特定属性

## 六、核心问题

**目标**: 设计一套通用解码器，语法能容纳标准卡组所有卡牌效果，实现模拟，不特殊化单卡，删除 rhea 功能集中在 abilities 中。

**关键矛盾**:
1. **通用性 vs 可实现性** — 完全形式化的卡牌效果语言需要解析器生成器，但项目倾向零正则
2. **覆盖率 vs 复杂度** — 覆盖 147 张遗漏卡需要更多 token/模式，增加维护负担
3. **统一 vs 特化** — 部分机制（corpse, kindred, colossal）有特殊规则，强行统一可能过度抽象
4. **abilities 系统的能力边界** — 适合"触发→效果"模式，但 aura/trigger 等持续效果不适合 parse-once
