# 外部研究资料: 卡牌效果解码器设计模式

> 检索日期: 2026-04-25

---

## 来源1: Fireplace — Python Hearthstone 模拟器 DSL

- URL: https://github.com/jleclanche/fireplace/wiki/2:-The-Fireplace-DSL
- 可信度: 高 (HearthSim 官方项目，Python 实现，已停止维护但设计经典)

### 关键设计

**DSL 核心概念**:
1. **Action Sequences** — 动作序列，可声明式定义（推荐）或 callable 实现
2. **Selectors** — 实体选择器，支持集合运算（+, -, |, ~），如 `FRIENDLY_MINIONS + BEAST`
3. **Card Pickers** — 随机卡牌选择，如 `RandomMinion(cost=5, rarity=LEGENDARY)`
4. **Evaluators** — 条件求值器，支持 then/else 子句：`Find(FRIENDLY_MINIONS + BEAST) & Hit(TARGET, 5) | Hit(TARGET, 3)`
5. **Lazy Numbers** — 延迟求值数字：`Count(selector)`, `Attr(selector, tag)`, `RandomNumber(n1, n2)`
6. **Event Arguments** — 事件回调中引用事件参数：`Death.ENTITY`

**优点**: 纯 Python 表达式作为 DSL，利用 Python 操作符重载（`&` = then, `|` = else, `*` = multiply）。声明式、可组合。
**缺点**: 需要为每张卡写 Python 代码，不支持从文本自动解析。

## 来源2: SabberStone — C# Hearthstone 模拟器 Task 模型

- URL: https://github.com/HearthSim/SabberStone/wiki/Implement-Cards
- 可信度: 高 (HearthSim 官方项目，C# 实现，活跃维护)

### 关键设计

**Power 模型**:
```csharp
public class Power {
    Aura Aura;           // 被动光环效果
    Enchant Enchant;     // 附魔（增益）
    Trigger Trigger;     // 触发器（Whenever/After）
    ISimpleTask PowerTask;      // 战吼/法术/英雄技能
    ISimpleTask DeathrattleTask; // 亡语
    ISimpleTask ComboTask;      // 连击
}
```

**ISimpleTask 任务链** — 用 `ComplexTask.Create(task1, task2, ...)` 组合：
- `IncludeTask(EntityType)` → 收集实体到栈
- `FilterStackTask(condition)` → 过滤栈中实体
- `CountTask(EntityType.STACK)` → 计数
- `DrawNumberTask()` → 按栈中数字抽牌
- `DamageTask(amount, target, spellDmg)` → 造成伤害
- `EnqueueTask(count, task, spellDmgAffectsCount)` → 重复执行

**Trigger** — 由 TriggerType（CAST_SPELL, TURN_END 等）+ TriggerSource + Condition + SingleTask 组成
**Aura** — 由 AuraType（BOARD, HAND 等）+ Effect/Enchantment + Condition 组成
**Enchantment** — 可自动从文本解析：`Enchants.GetAutoEnchantFromText(cardId)` 支持 `+n/+m`, `+n Attack`, 关键词等

**优点**: Task 链模型非常通用，任何效果都可以用任务序列表达。每张卡的实现是声明式的。
**缺点**: 仍需为每张卡手写 C# 代码。

## 来源3: Reddit r/gamedev — 卡牌效果设计讨论

- URL: https://www.reddit.com/r/gamedev/comments/1g2grgp/whats_the_right_way_to_code_tons_of_card_effects/
- 可信度: 中 (社区讨论，多种方案)

### 关键方案

1. **状态机执行** — 每个效果条目存储类型+触发时机+参数，执行时启动状态机实例
2. **ECS (Entity Component System)** — 用组件动态构造卡牌，新类型不需要修改现有代码
3. **DSL/脚本语言** — JSON/Lua 定义卡牌，解释器执行
4. **数据驱动** — 固定属性 + 效果列表，每个效果有类型、触发时机、基础值和等级增量

### 共识

- 避免为每张卡写独立逻辑（hardcode）
- 用数据格式定义卡牌，引擎解释执行
- 复杂卡用脚本语言（DSL/Lua），简单卡用数据格式（JSON）

## 综合发现

### 设计模式对比

| 维度 | Fireplace DSL | SabberStone Tasks | 本项目 abilities |
|------|--------------|-------------------|------------------|
| **语言** | Python 表达式 | C# 任务链 | Python keyword table |
| **定义方式** | 每卡手写代码 | 每卡手写代码 | 从文本自动解析 |
| **效果表达** | Action 对象 + 选择器 | ISimpleTask 链 | EffectKind 枚举 |
| **条件** | Evaluator 对象 | SelfCondition | ConditionKind 枚举 |
| **目标** | Selector 集合运算 | EntityType 枚举 | TargetKind 枚举 |
| **触发器** | Event listener | Trigger 对象 | AbilityTrigger 枚举 |
| **光环** | 内置 Aura | Power.Aura 属性 | 独立 aura_engine |
| **延迟求值** | LazyNum | 栈传递 | 无 |
| **覆盖率** | 100%（手写） | 100%（手写） | 60.8%（自动解析） |

### 核心洞察

1. **所有成熟模拟器都手写每张卡** — Fireplace 和 SabberStone 都不为每张卡自动解析文本，而是用 DSL/Task 声明式描述。这是最可靠的方案。
2. **自动解析的极限** — 本项目 abilities parser 的 60.8% 覆盖率说明，纯文本匹配无法覆盖所有效果，特别是条件分支、延迟求值、复杂目标选择。
3. **关键缺失能力** — 延迟求值（LazyNum/栈传递）和条件分支（if-then-else）是 abilities 系统最大的差距。
4. **"不特殊化单卡"的矛盾** — 真正通用的系统要么手写每张卡（Fireplace/SabberStone），要么从结构化数据自动生成。从非结构化文本自动解析到 100% 覆盖率在实践中不可行。
