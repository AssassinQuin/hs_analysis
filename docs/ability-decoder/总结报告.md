# 能力解码器 (Ability Decoder) 调研总结报告

> 生成日期: 2026-04-25
> 研究范围: analysis/search/rhea/, analysis/search/abilities/, 标准卡池 984 张卡, Fireplace/SabberStone 外部参考

## 一、研究背景与目标

### 1.1 问题描述

当前项目有两套卡牌效果处理系统并行运行：

1. **abilities 系统**（`analysis/search/abilities/`，923 行）— 基于关键词表和文本匹配的自动解析器，parse-once-execute-many 模式
2. **spell_simulator + battlecry_dispatcher**（~1,257 行）— 基于 regex 的独立效果解析和执行系统

两套系统功能大量重叠（伤害、召唤、抽牌、治疗等），但都不完整：
- abilities 覆盖率仅 60.8%（598/984 张标准卡）
- spell_simulator 仅用于法术，battlecry_dispatcher 仅用于战吼
- 亡语已迁移到 abilities，但战吼和法术仍用旧系统
- 25+ 个独立机制模块（corpse, kindred, colossal 等）各有特定逻辑

此外，`analysis/search/rhea/` 模块（1,171 行）的 RHEA 搜索引擎已删除，剩余的 `actions.py`、`enumeration.py`、`simulation.py` 是 MCTS 的共享原语，命名上已不合时宜。

### 1.2 目标

1. **设计通用解码器** — 一套语法体系能容纳当前标准环境所有 984 张卡的效果
2. **消除特殊化** — 不为某张卡专门设计功能（如 `check_herald()`、`apply_kindred()`），所有效果通过通用机制表达
3. **合并到 abilities** — 将 rhea 的 simulation/enumeration 功能和 spell_simulator/battlecry_dispatcher 的效果执行全部整合到 abilities 模块中
4. **删除 rhea 目录** — 相关功能迁移后删除 `analysis/search/rhea/`

### 1.3 约束

- **不能破坏 MCTS 引擎** — MCTS 的 `engine.py`、`simulation.py`、`expansion.py` 等大量导入 rhea 模块
- **保留 GameState 不变** — GameState 是核心数据结构，不能因为解码器重构而改接口
- **渐进式迁移** — 不能一次性替换所有模块，需要保持测试通过
- **不追求 100% 自动解析** — 参考 Fireplace/SabberStone，成熟模拟器都手写每张卡。自动解析器目标是覆盖常见模式，特殊卡允许注册自定义处理器

## 二、现有项目基础

### 2.1 abilities 系统 (`analysis/search/abilities/`, 923 行)

**设计**: parse-once, execute-many。CardAbility 对象缓存在 Card model 的 lazy `abilities` 属性上。

**类型系统** (`definition.py`, 164 行):
- `AbilityTrigger`(22 值) — BATTLECRY, DEATHRATTLE, TRIGGER_VISUAL, TURN_END, WHENEVER, ON_SPELL_CAST 等
- `EffectKind`(21 值) — DAMAGE, SUMMON, DRAW, GAIN, GIVE, HEAL, DESTROY, DISCOVER 等
- `ConditionKind`(8 值) — HOLDING_RACE, THIS_TURN, FOR_EACH, HAS_KEYWORD 等
- `TargetKind`(13 值) — SINGLE_MINION, RANDOM, ALL_ENEMY, ALL_FRIENDLY 等
- 4 个 dataclass: `CardAbility`, `EffectSpec`, `TargetSpec`, `ConditionSpec`

**映射表** (`tokens.py`, 144 行):
- `MECHANICS_TRIGGER_MAP`(13 项) — mechanics tag → AbilityTrigger
- `STATIC_KEYWORD_MECHANICS`(17 项) — TAUNT, RUSH, LIFESTEAL 等不产生 ability 的静态关键词
- `ACTION_VERB_MAP`(21 项) — 英文动词 → EffectKind
- `TEXT_TRIGGER_MAP`(10 项) — 文本触发器短语

**解析器** (`parser.py`, 197 行): 两阶段解析
1. mechanics tags → MECHANICS_TRIGGER_MAP → CardAbility
2. English text → TEXT_TRIGGER_MAP + ACTION_VERB_MAP → CardAbility

**执行器** (`executor.py`, 236 行):
- `trigger(state, event, source, target)` — 遍历实体，匹配触发器，执行效果
- 已实现: WEAPON_EQUIP, DAMAGE, SUMMON, DRAW, GAIN, HEAL, GIVE
- 空操作: DISCOVER, DESTROY, FREEZE, SILENCE

### 2.2 rhea 模块 (`analysis/search/rhea/`, 1,171 行)

RHEA 搜索引擎已删除（commit a82b89f），剩余为 MCTS 共享原语：

| 文件 | 行数 | 职责 | 消费者 |
|------|------|------|--------|
| `actions.py` | 100 | ActionType 枚举 (10 值) + Action dataclass | MCTS, engine/pipeline, action_pruner 等 20+ 文件 |
| `enumeration.py` | 297 | `enumerate_legal_actions(state)` | MCTS expansion, engine/tactical |
| `simulation.py` | 749 | `apply_action(state, action)` | MCTS simulation, engine/pipeline |

**simulation.py 的角色**: 它是**编排层**，不实现具体效果，而是调用 25+ 个机制模块：
- battlecry_dispatcher → 战吼分发
- deathrattle → 亡语解析（已用 abilities）
- spell_simulator.resolve_effects → 法术效果
- aura_engine → 光环重算
- colossal/corpse/corrupt/dormant/herald/imbue/kindred/location/outcast/quest/shatter → 特殊机制
- trigger_system → 触发器分发
- abilities → ON_SPELL_CAST, PASSIVE_COST, WHENEVER 等

**核心调用链**: `enumerate_legal_actions()` → `[Action]` → MCTS 选择一个 → `apply_action()` → 委托给机制模块 → 新 GameState

### 2.3 独立效果系统

**spell_simulator** (`analysis/utils/spell_simulator.py`, 753 行):
- `EffectParser.parse(card)` → 从 CardEffects 结构化数据 + text fallback regex 解析效果
- `EffectApplier.apply(state, effects, target)` → 执行效果（aoe_damage, random_damage, direct_damage, summon, draw, destroy, heal, armor, buff, discard, cost_reduce）
- **问题**: 与 abilities executor 功能重叠，用 regex 而非 abilities 的 keyword table

**battlecry_dispatcher** (`analysis/search/battlecry_dispatcher.py`, 504 行):
- regex 提取战吼文本
- 9 种战吼子类型检查（destroy_minion, freeze, silence, give_divine_shield, give_taunt 等）
- 复用 spell_simulator 的 EffectParser + EffectApplier
- **问题**: 每种战吼子类型都是硬编码正则，与 abilities 的 ACTION_VERB_MAP 重叠

### 2.4 已迁移到 abilities 的模块

**deathrattle.py** (141 行) — 先查 enchantment 触发器，再查 abilities DEATHRATTLE trigger。是成功迁移的范例。
**location.py** — 用 AbilityParser.parse(loc) 获取 ACTIVATE 触发的效果。

### 2.5 保持独立的模块

| 模块 | 原因 |
|------|------|
| aura_engine | 光环需要持续重算，不适合 parse-once 模型 |
| trigger_system | 触发器注册和分发是运行时行为 |
| discover.py | 发现池计算是数据查询 |
| game_state.py | 核心数据结构 |
| colossal/corpse/dormant/herald/imbue/kindred/outcast/quest/shatter | 各有特定规则（见 §四分析） |

## 三、外部研究综合

### 3.1 Fireplace — Python DSL 模式 [E1][E2]

Fireplace 是 Python 实现的 Hearthstone 模拟器，使用 Python 表达式作为 DSL：

```python
# Kill Command: 如果场上有野兽则 5 伤害，否则 3 伤害
Find(FRIENDLY_MINIONS + BEAST) & Hit(TARGET, 5) | Hit(TARGET, 3)

# Battle Rage: 每个受伤友方角色抽一张牌
Draw(CONTROLLER) * Count(FRIENDLY_MINIONS + DAMAGED)
```

**核心设计**:
- **Selector** — 实体选择器，支持集合运算 (`+` 交集, `|` 并集, `-` 差集)
- **Action** — 原子操作（Hit, Draw, Summon, Give, Destroy 等）
- **Evaluator** — 条件求值，`&` = then, `|` = else
- **LazyNum** — 延迟求值数字（Count, Attr, RandomNumber），用 `*` 乘法连接
- 每张卡手写 Python 表达式，不用自动文本解析

### 3.2 SabberStone — C# Task 链模式 [E3]

SabberStone 用 `Power` 类描述卡牌效果，`ISimpleTask` 链执行：

```csharp
// Battle Rage: 抽等于受伤友方角色数量的牌
PowerTask = ComplexTask.Create(
    new IncludeTask(EntityType.FRIENDS),       // 收集所有友方实体
    new FilterStackTask(SelfCondition.IsDamaged), // 过滤受伤的
    new CountTask(EntityType.STACK),            // 计数
    new DrawNumberTask())                       // 按数量抽牌
```

**核心设计**:
- **Power** — 卡牌效果容器（PowerTask, DeathrattleTask, Aura, Trigger, Enchant）
- **ISimpleTask** — 原子任务，用 ComplexTask.Create 串联
- **栈传递** — 任务间通过栈共享中间结果（Include → Filter → Count → Draw）
- **Trigger** — TriggerType + TriggerSource + Condition + SingleTask
- **Aura** — AuraType + Effect + Condition，支持 AdaptiveEffect
- **Enchantment** — 可从文本自动解析（`+n/+m`, `+n Attack`, 关键词）

### 3.3 关键洞察

| 洞察 | 详情 |
|------|------|
| **所有成熟模拟器都手写每张卡** | Fireplace 用 Python DSL，SabberStone 用 C# Task。没有项目实现 100% 自动文本解析 |
| **通用原语集** | 两个项目都定义了 ~50-80 种原子操作（Damage, Draw, Summon, Buff 等），足够覆盖所有卡牌效果 |
| **延迟求值是关键** | Count(selector) 和条件分支（if-then-else）是表达复杂效果的必要能力，当前 abilities 系统缺失 |
| **Enchantment 可自动解析** | SabberStone 的 `GetAutoEnchantFromText()` 能从 `+n/+m` 文本自动生成 buff，本项目的 abilities 提取器已有类似能力 |
| **特殊机制需要注册器** | SabberStone 的每张卡是一个"注册"，不是硬编码的 if/elif。特定机制（如 Corpse、Colossal）仍需专属 Task |

## 四、核心问题分析

### 4.1 覆盖率缺口 (147/984 张卡)

abilities parser 覆盖 598/984 (60.8%)，遗漏 147 张有 mechanics 的卡：

| 遗漏类别 | 数量 | 根因 | 严重程度 |
|----------|------|------|----------|
| 静态关键词组合（仅 TAUNT+RUSH 等） | ~70 | `STATIC_KEYWORD_MECHANICS` 被排除，不产生 ability | 🟡 中 — 静态关键词在模拟中已通过 GameState 属性处理 |
| 仅 DISCOVER | 33 | DISCOVER 不在 MECHANICS_TRIGGER_MAP 中 | 🔴 高 — 33 张法术完全无效果模拟 |
| 仅 AURA | 16 | AURA 不在 MECHANICS_TRIGGER_MAP 中 | 🟡 中 — AURA 由独立 aura_engine 处理 |
| 仅 OVERLOAD | 10 | OVERLOAD 不在 MECHANICS_TRIGGER_MAP 中 | 🟡 中 — simulation.py 单独处理 overload |
| QUEST/SIDE_QUEST | 9 | QUEST 解析不完整 | 🟡 中 — quest.py 独立处理 |
| TRIGGER_VISUAL 无文本匹配 | 9 | TEXT_TRIGGER_MAP 覆盖不全 | 🔴 高 — 触发效果无模拟 |

### 4.2 双系统重叠

| 效果 | abilities executor | spell_simulator | battlecry_dispatcher |
|------|-------------------|-----------------|---------------------|
| DAMAGE | ✅ `DAMAGE` | ✅ `direct_damage/aoe_damage/random_damage` | ✅ 委托 spell_sim |
| SUMMON | ✅ `SUMMON` | ✅ `summon/summon_stats` | — |
| DRAW | ✅ `DRAW` | ✅ `draw` | — |
| HEAL | ✅ `HEAL` | ✅ `heal` | — |
| GAIN (armor) | ✅ `GAIN` | ✅ `armor` | — |
| GIVE (buff) | ✅ `GIVE` | ✅ `hand_buff/buff_atk` | ✅ regex 匹配 |
| DESTROY | ❌ 空操作 | ✅ `destroy` | ✅ regex 匹配 |
| DISCOVER | ❌ 空操作 | — | ✅ regex 匹配 |
| FREEZE | ❌ 空操作 | — | ✅ regex 匹配 |
| SILENCE | ❌ 空操作 | — | ✅ regex 匹配 |
| DISCARD | — | ✅ `discard` | — |
| COST_REDUCE | — | ✅ `cost_reduce` | — |

**结论**: abilities executor 覆盖了 7 种效果，spell_simulator 额外覆盖 3 种，battlecry_dispatcher 又额外覆盖 4 种。三套系统合起来才完整，但任何单套都不完整。

### 4.3 静态关键词的处理矛盾

17 个 `STATIC_KEYWORD_MECHANICS`（TAUNT, RUSH, LIFESTEAL, CHARGE, DIVINE_SHIELD 等）不产生 ability，因为它们：
- 是实体属性，不是触发效果
- 在 enumeration.py（TAUNT 影响攻击目标选择）和 simulation.py（DIVINE_SHIELD 吸收伤害，LIFESTEAL 回血等）中直接处理

但 70+ 张仅有这些关键词的卡在 abilities 系统中完全透明，无法通过 abilities 系统查询"这张卡有什么效果"。

### 4.4 延迟求值和条件分支缺失

当前 abilities 系统的 ConditionSpec 只有 8 种条件（HOLDING_RACE, THIS_TURN, FOR_EACH 等），无法表达：
- **"Draw a card for each damaged friendly character"** — 需要 `Count(DAMAGED_FRIENDS) → Draw(that_many)`
- **"If you're holding a Dragon, deal 5 damage; otherwise, deal 3"** — 需要 if-then-else 分支
- **"Deal damage equal to your hero's Attack"** — 需要 `Attr(HERO, ATK)` 延迟求值
- **"Summon a random minion with cost equal to your hero's Attack"** — 需要交叉引用属性

SabberStone 用栈传递解决这个问题，Fireplace 用 LazyNum。当前 abilities 没有类似机制。

### 4.5 机制模块的特殊性分析

哪些机制可以通过通用解码器表达，哪些需要专属逻辑：

| 机制模块 | 能否通用化 | 原因 |
|----------|-----------|------|
| **battlecry** | ✅ 完全可以 | 就是 BATTLECRY trigger + 效果序列 |
| **deathrattle** | ✅ 已迁移 | 证明了 abilities 模式可行 |
| **spell effects** | ✅ 完全可以 | 就是 PLAY trigger + 效果序列 |
| **hero power** | ✅ 完全可以 | 就是 ACTIVATE trigger |
| **location** | ✅ 已迁移 | ACTIVATE trigger + 效果序列 |
| **aura** | ⚠️ 部分 | 被动光环需要持续重算，不适合 parse-once；但 buff 类光环可以用 GIVE 效果 |
| **trigger (whenever/after)** | ⚠️ 部分 | 触发器注册是运行时行为，但触发效果的执行可以通用化 |
| **colossal** | ❌ 需要专属 | 召唤附属随从需要卡池查询 + 特定位置规则 |
| **corpse** | ❌ 需要专属 | 资源系统，不是效果类型 |
| **dormant** | ❌ 需要专属 | 休眠倒计时是独立状态机 |
| **kindred** | ❌ 需要专属 | "施放两次"需要复制整次出牌过程 |
| **imbue** | ❌ 需要专属 | 灌注计数器是独立状态 |
| **quest** | ❌ 需要专属 | 任务进度追踪是独立系统 |
| **outcast** | ⚠️ 可条件化 | "如果是最左/右牌" → 条件检查 |
| **corrupt** | ⚠️ 可条件化 | "如果手牌中有更高费的卡" → 条件检查 |
| **shatter** | ⚠️ 可条件化 | "如果对手被冻结" → 条件检查 |

### 5.2 核心扩展：LazyValue 和条件分支

**新增 `LazyValue`** — 延迟求值数字，灵感来自 Fireplace LazyNum：

```python
# definition.py 扩展
class LazyValue:
    """延迟求值 — 运行时根据 GameState 计算实际数值"""
    def evaluate(self, state, source) -> int: ...

class CountEntities(LazyValue):
    """Count(selector) — 计算匹配选择器的实体数量"""
    def __init__(self, selector: EntitySelector): ...
    def evaluate(self, state, source) -> int:
        return len(selector.select(state, source))

class AttrValue(LazyValue):
    """Attr(selector, attr) — 获取实体属性值"""
    def __init__(self, selector: EntitySelector, attr: str): ...
    def evaluate(self, state, source) -> int:
        entities = selector.select(state, source)
        return sum(getattr(e, attr, 0) for e in entities)

class LiteralValue(LazyValue):
    """固定数值"""
    def __init__(self, value: int): ...
    def evaluate(self, state, source) -> int:
        return self.value
```

**扩展 `EffectSpec`** — value 字段支持 LazyValue：

```python
@dataclass
class EffectSpec:
    kind: EffectKind
    value: int | LazyValue = 0     # ← 允许延迟求值
    value2: int | LazyValue = 0
    subtype: str | None = None
    keyword: str | None = None
    target: TargetSpec | None = None
    text_raw: str = ""
```

**新增 `ConditionalEffect`** — 条件分支：

```python
@dataclass
class ConditionalEffect:
    """If condition then effects_true else effects_false"""
    condition: ConditionSpec
    if_effects: list[EffectSpec]
    else_effects: list[EffectSpec] | None = None
```

### 5.3 核心扩展：EntitySelector

**新增 `EntitySelector`** — 统一的实体选择器，替代 TargetKind 枚举：

```python
class EntitySelector:
    """实体选择器 — 灵感来自 Fireplace Selector"""
    def select(self, state, source) -> list: ...

class AllMinions(EntitySelector): ...
class FriendlyMinions(EntitySelector): ...
class EnemyMinions(EntitySelector): ...
class Damaged(EntitySelector): ...          # 包装器：仅受伤的
class HasRace(EntitySelector, race: str): ...  # 包装器：特定种族
class RandomN(EntitySelector, n: int): ...  # 随机 N 个
class HeroSelector(EntitySelector): ...

# 组合操作
selector = FriendlyMinions() + HasRace("DRAGON")  # 交集
selector = EnemyMinions() | HeroSelector()          # 并集
selector = AllMinions() - FriendlyMinions()         # 差集 = 敌方随从
```

### 5.4 机制注册器（MechanicRegistry）

特殊机制不通过 abilities parser 解析，而是注册为**模拟钩子**：

```python
class MechanicHook:
    """机制钩子 — 在特定时机触发的特殊逻辑"""
    trigger: str              # "pre_play", "post_play", "on_death", "end_turn"
    card_filter: Callable     # 判断是否适用（如 mechanics 包含 COLOSSAL）
    handler: Callable         # 处理函数

class MechanicRegistry:
    _hooks: list[MechanicHook] = []

    @classmethod
    def register(cls, trigger, card_filter, handler): ...

    @classmethod
    def fire(cls, trigger, state, card, **kwargs) -> GameState:
        for hook in cls._hooks:
            if hook.trigger == trigger and hook.card_filter(card):
                state = hook.handler(state, card, **kwargs)
        return state
```

注册示例：
```python
# colossal.py 注册为钩子，不再是 simulation.py 的显式 import
MechanicRegistry.register(
    trigger="post_summon",
    card_filter=lambda c: "COLOSSAL" in c.mechanics_set,
    handler=summon_colossal_appendages
)
```

### 5.5 合并策略

**Step 1: rhea 模块迁入 abilities**

| 文件 | 来源 | 目标 |
|------|------|------|
| `actions.py` | `rhea/actions.py` | `abilities/actions.py` — 不改代码 |
| `enumeration.py` | `rhea/enumeration.py` | `abilities/enumeration.py` — 不改代码 |
| `simulation.py` | `rhea/simulation.py` | `abilities/simulation.py` — 逐步替换机制模块调用 |

`rhea/__init__.py` 变成 re-export shim，所有消费者不受影响。

**Step 2: 合并效果执行**
