# MCTS/UCT 炉石传说搜索算法 — 调研总结报告

> 生成日期: 2026-04-24
> 研究范围: 基于 hs_analysis 项目现有架构，设计并实现 MCTS/UCT 搜索引擎

---

## 一、研究背景与目标

### 1.1 问题定义

hs_analysis 项目当前的搜索引擎是 RHEA (Rolling Horizon Evolutionary Algorithm)，它通过进化算法优化动作序列。虽然 RHEA 在实践中有一定效果，但存在以下局限：

1. **全局搜索能力不足**: RHEA 是种群级搜索，难以保证对动作空间的全局覆盖
2. **无法自然处理不确定性**: 对手手牌、牌库、奥秘等不完全信息需要额外确定化层
3. **缺乏理论收敛保证**: 进化算法不像 MCTS 有 UCT 的 Hoeffding 不等式收敛保证
4. **动作空间探索不充分**: 大动作空间下 RHEA 容易陷入局部最优

MCTS/UCT 作为经典的不完全信息博弈搜索算法，在炉石传说领域已有充分的学术验证 [S1][S2][S3]，是 RHEA 的有力替代或补充。

### 1.2 目标

设计一个 MCTS/UCT 搜索引擎，满足：
- **兼容现有架构**: 复用 GameState、动作空间、评估函数等核心组件
- **集成到决策管线**: 与致命检测、风险评估等现有层协同工作
- **适应时间预算**: 在 3-15 秒内完成搜索并输出最优动作序列
- **处理不完全信息**: 通过确定化 (DUCT) 处理对手隐藏信息
- **大规模动作空间**: 通过渐进展开和动作剪枝处理高分支因子

## 三、外部研究综合

### 3.1 确定化 UCT (DUCT) — 处理不完全信息的主流方案

据 Zhang (2017) [S3] 和 Choe & Kim (2020) [S1] 的研究，DUCT 是处理炉石不完全信息最实用的方案：

**核心思路**:
1. 预采样 N 个"世界" (determinizations)，每个世界包含对手手牌和牌库的完整假设
2. 每次迭代选择一个世界，在完美信息下执行 MCTS
3. 多个世界共享相同动作时，聚合统计用于 UCB1 选择
4. 最终选择根节点访问次数最多的动作

**参数经验值** [S3]:
- numWorlds: 5-11 (随对手信息不确定性增加)
- UCT 常数 c: 0.25-1.0 (因卡组而异)
- Rollout 深度: 3 回合

**胜率验证** [S3]: DUCT vs Silverfish — 57-68% 胜率

### 3.2 大动作空间处理

炉石单回合动作空间可达数千。据多项研究 [S1][S3][S7]，以下方法有效：

| 方法 | 来源 | 效果 |
|------|------|------|
| 渐进展开 (Progressive Widening) | [S1][S7] | 节点访问次数 n^α 控制子节点数 |
| 动作过滤 (Category-based Filtering) | [S1] | 移除 30-50% 自毁/浪费动作 |
| 必要动作 (Obliged Actions) | [S1] | 无条件有利动作强制执行 |
| 机会节点分桶 (CNB) | [S3] | 随机事件分支减少 50% |
| 软等价 (Soft Hash) | [S3] | 聚合等价动作统计 |

### 3.3 DAG 搜索 + 转置表

Choe & Kim [S1] 提出 UCD 算法将搜索空间表示为 DAG：
- 统计信息存储在**边**上而非节点上
- 状态抽象将相似状态聚合为单一节点
- 效果：搜索空间显著减小，尤其在同回合内动作可交换的场景

Świechowski & Tajmajer [S2] 提出多层转置表：
- IS-state (信息集) 作为键 → MCTS 节点
- 同一信息集对应多个 GS-state 的统计
- 完全消除机遇节点，简化树结构

### 3.4 Rollout 策略

据 Zhang (2017) [S3] 和 peter1591 [P1] 的实践：

**方案 A — 启发式评估截断** (推荐):
- 不执行完整游戏模拟
- 使用评估函数 (如 Silverfish 或本项目 FactorGraphEvaluator) 直接评估叶节点
- 优点: 速度快，无需完整模拟
- 缺点: 评估函数质量决定上限

**方案 B — 随机 Rollout + 动作过滤**:
- 随机走子但过滤明显坏动作
- 深度限制 2-3 回合
- 优点: 接近 MCTS 理论保证
- 缺点: Python 速度慢，有效迭代次数少

**方案 C — 策略网络引导** [S3][P1]:
- CNN 预测动作概率作为 rollout policy
- 优点: 高质量 rollout
- 缺点: 需要训练数据和模型

### 3.5 时间预算策略

据 [S1][S3] 的实验：
- **指数递减** (Choe & Kim): `T(n) = T0 * γ^{n-1}` — 效果显著 [S1]
- **分支因子自适应** (Zhang): `LB(n) = τ * T / BF(n)` — 对复杂局面更稳健 [S3]
- **混合策略**: 先做一次完整搜索，再对弱访问节点补充搜索 [S3]

## 四、方案设计

### 4.1 推荐架构: 分层 DUCT

基于文献研究和项目约束，推荐 **分层 DUCT + 评估截断** 方案：

```
决策管线 (复用现有):
  Layer 0: 致命检测 (5ms, 复用)
      ↓ 未找到致命
  Layer 1: MCTS/UCT 搜索引擎 (主搜索层)
      ↓ 输出最优动作序列
  Layer 2: 风险评估 + 对手模拟 (复用)
      ↓ 安全验证
  输出: 最终动作序列
```

### 4.2 MCTS 核心组件设计

#### 4.2.1 树节点设计

```python
@dataclass
class MCTSNode:
    """MCTS 树节点"""
    state_hash: int           # IS-state 哈希 (用于转置表)
    parent: Optional['MCTSNode']
    children: Dict[int, 'MCTSNode']  # action_key → child
    
    # 统计信息
    visit_count: int = 0
    total_reward: float = 0.0
    
    # 动作相关
    untried_actions: List[Action] = field(default_factory=list)
    action_edges: Dict[int, 'ActionEdge'] = field(default_factory=dict)
    
    # 上下文
    is_player_turn: bool = True
    phase: Phase = Phase.MID
```

#### 4.2.2 动作边设计

```python
@dataclass
class ActionEdge:
    """MCTS 边 — 存储动作统计"""
    action: Action
    child_node: Optional[MCTSNode] = None
    
    # 边统计
    visit_count: int = 0
    total_reward: float = 0.0
    
    # UCT 辅助
    progressive_widening_threshold: int = 0
```

#### 4.2.3 搜索流程

```
MCTS 主循环:
  1. 确定化: 从信息集采样 numWorlds 个世界
  2. 循环 (直到时间预算用尽):
     a. 选择世界 w
     b. Selection: 从根节点沿 UCT 最优边下降
     c. Expansion: 展开一个未尝试动作
     d. Simulation: 评估截断 (调用 evaluate_delta)
     e. Backpropagation: 沿路径更新统计
  3. 从根节点选择访问次数最多的动作
  4. 以该动作后的状态为新根，重复搜索下一个动作
  5. 拼接所有选择的动作 → 完整动作序列
```

### 4.3 关键算法参数

| 参数 | 推荐值 | 来源 |
|------|--------|------|
| UCT 常数 c | 0.5 (初始), 可按卡组调优至 0.25-1.0 | [S1][S3] |
| numWorlds (确定化数) | 7 (默认), 5-11 可调 | [S3] |
| 渐进展开 α | 0.5 | [S1] |
| 渐进展开 C | 1.0 | [S1] |
| Rollout 策略 | 评估截断 (evaluate_delta) | 项目复用 |
| 时间预算 | 总 5-10s, 指数递减 T0=3s, γ=0.6 | [S1] |
| 最大迭代次数 | 时间限制为主 | — |
| 动作过滤 | 启用 (Category-based + Obliged) | [S1] |

### 4.4 方案对比

| 维度 | 方案 A: DUCT+评估截断 | 方案 B: DUCT+随机Rollout | 方案 C: DAG-UCD |
|------|----------------------|-------------------------|----------------|
| 实现复杂度 | ★★☆ 中等 | ★☆☆ 简单 | ★★★ 复杂 |
| 搜索效率 | ★★★ 高 (Python友好) | ★☆☆ 低 (Python慢) | ★★★ 高 |
| 理论保证 | ★★☆ 弱 (截断近似) | ★★★ 强 (接近理论) | ★★★ 强 |
| 扩展性 | ★★★ 好 | ★★☆ 中 | ★★☆ 中 |
| 推荐度 | **推荐** | 不推荐 (Python太慢) | 可选 (需更多工作) |

### 4.5 性能预估

基于 Python 速度和文献数据：
- 单次 MCTS 迭代 (Selection + Expansion + Eval): ~1-5ms
- 5 秒时间预算: ~1000-5000 次迭代
- 7 个世界 × 700 次迭代/世界
- 对比 RHEA (50-200 染色体评估): MCTS 提供更广泛的空间覆盖

### 4.6 Python 性能优化策略

1. **评估截断替代完整 Rollout**: 省去最耗时的模拟步骤
2. **转置表 + 状态哈希**: 避免重复展开相同状态
3. **动作预计算缓存**: enumerate_legal_actions 结果缓存
4. **渐进展开**: 不一次性展开所有子节点
5. **轻量状态复制**: 仅在展开时复制，Selection 阶段不复制

## 五、实施建议

### 5.1 开发路线图

| 阶段 | 内容 | 预计工作量 |
|------|------|-----------|
| **P0: 基础 MCTS** | 树节点、UCT 选择、基本搜索循环 | 3-5 天 |
| **P1: 评估集成** | 接入 evaluate_delta 作为 rollout | 1-2 天 |
| **P2: 确定化层** | DUCT 多世界采样 + BayesianOpponent | 2-3 天 |
| **P3: 动作剪枝** | Category-based filtering + Obliged actions | 2-3 天 |
| **P4: 渐进展开** | Progressive Widening 集成 | 1-2 天 |
| **P5: 管线集成** | 替换 RHEA 或作为并行搜索选项 | 2-3 天 |
| **P6: 参数调优** | UCT c, numWorlds, 时间分配等 | 3-5 天 |
| **P7: 测试验证** | 单元测试 + 对战 RHEA A/B 测试 | 3-5 天 |

### 5.2 文件结构建议

```
analysis/search/mcts/
├── __init__.py
├── engine.py          # MCTSEngine 主类
├── node.py            # MCTSNode, ActionEdge
├── uct.py             # UCT 选择策略
├── expansion.py       # 展开策略 + 渐进展开
├── simulation.py      # Rollout/评估截断策略
├── backprop.py        # 反向传播
├── determinization.py # DUCT 确定化采样
├── pruning.py         # 动作过滤 + Obliged actions
├── transposition.py   # 转置表
└── config.py          # MCTS 参数配置
```

### 5.3 关键风险

1. **Python 性能瓶颈**: MCTS 需要大量迭代，Python 可能无法在时间预算内完成足够搜索
   - 缓解: 评估截断 + 动作剪枝 + 渐进展开减少单次迭代开销
   
2. **GameState.copy() 开销**: 每次展开需要复制完整状态
   - 缓解: 只在展开时复制，探索路径复用状态引用
   
3. **评估函数偏差**: 截断 rollout 的质量完全依赖评估函数
   - 缓解: 可选混合策略 — 短 rollout (1-2步) + 评估

4. **与 RHEA 的集成**: 两个搜索引擎如何共存
   - 建议: 管线中加入 MCTS 层，或作为可切换选项
