## 一、现有实现

### 1.1 搜索引擎架构

- [analysis/search/rhea/engine.py] — RHEAEngine，当前主要搜索引擎 (L64-913)
  - 滚动视野进化算法 (Rolling Horizon Evolutionary Algorithm)
  - 5层决策管线: Lethal(5ms) → UTP(10%) → RHEA(50%) → Multi-turn(10%) → Cross-turn(20%)
  - 阶段自适应参数: EARLY(pop30/gen100/len4), MID(pop50/gen200/len6), LATE(pop60/gen150/len8)
  - 适应时间预算: 3-5s (正常) / 5-15s (复杂)

- [analysis/search/engine/unified_tactical.py] — UTP 束搜索 (Beam Search, 124行)
  - FactorGraphEvaluator 多维度评估

- [analysis/search/lethal_checker.py] — DFS 致命检测 (280行)
  - `max_damage_bound()` 上界剪枝
  - `check_lethal()` 5ms 预算内穷举

- [analysis/search/opponent_simulator.py] — 贪心1回合对手模拟 (180行)
  - `simulate_best_response()` 对手最佳响应

- [analysis/search/risk_assessor.py] — 风险评估 (113行)

## 三、经验与约束

### 设计决策
- 状态使用 `copy()` 不可变模式 → MCTS 需要大量节点复制，内存是瓶颈
- 动作通过 `action_key()` 哈希 → 可用于 MCTS 节点的子节点索引
- 模拟支持优雅降级 (`_try_mechanic`) → MCTS rollout 中可选择性跳过复杂机制
- 评估函数已有阶段感知权重 → MCTS rollout 可直接复用
- 已有对手模型 (Bayesian) → 可驱动确定化采样

### 已知限制
- Python 性能瓶颈: 状态复制和动作模拟速度是 C++ 的 ~1/100
- 大动作空间: 完整回合可达百万级分支
- 不完全信息: 对手手牌/牌库/奥秘未知，需要确定化策略
- 机制复杂度: 24+ 机制模块，rollout 中全模拟太慢

### 架构约束 (来自 config.py)
- 搜索引擎通过 DecisionLoop → StateBridge → Engine 调用
- 入口点: scripts/run_live.py, scripts/run_rhea.py
- 需要与现有分层管线兼容 (Lethal → Search → Risk → Opponent Sim)

## 四、知识空白

1. **MCTS 节点设计**: 如何在 GameState 之上设计高效的 MCTS 树节点 (内存、哈希、缓存)
2. **UCT 参数调优**: 炉石场景下 UCB1 探索常数 C 的最优值
3. **渐进展开 (Progressive Widening)**: 大动作空间下的具体参数和策略
4. **Rollout 策略**: 如何设计轻量但有效的快速走子策略 (不全部模拟机制)
5. **确定化采样**: 不完全信息下采样数量和聚合方法
6. **时间预算分配**: MCTS 各阶段的时间分配 vs RHEA 的比较
7. **RHEA vs MCTS 性能对比**: 在相同基础设施上的胜率比较
8. **并行化**: MCTS 的并行搜索 (Tree Parallelization) 在 Python 中的实现方案
9. **回合级别搜索**: 炉石中一个"回合"是多步动作序列，如何映射到 MCTS 树结构
10. **终止条件**: 单回合 MCTS 的终止判断 (END_TURN 作为终止动作的处理)
