## Batch 1: Kindred + Corpse（无交叉依赖，可并行）

### MT-1.1: GameState 扩展 — corpses + kindred_double_next

**目标**: 在 GameState 添加残骸资源和延系加倍标志位

**修改文件**: `hs_analysis/search/game_state.py`

**关键实现点**:
- GameState 新增 `corpses: int = 0` 字段
- GameState 新增 `kindred_double_next: bool = False` 字段
- 这两个字段需要在 `copy()` / `deepcopy()` 中正确传播

**验证**: 运行现有 game_state 相关测试确认无回归

### MT-1.2: kindred.py — 延系机制

**目标**: 实现延系条件检测 + 奖励效果解析 + 应用

**新建文件**: `hs_analysis/search/kindred.py`

**关键实现点**:
1. `has_kindred(card_text: str) -> bool` — 检测 `<b>延系</b>` 或 `<b>延系：</b>`
2. `parse_kindred_bonus(card_text: str) -> str | None` — 提取延系后的效果描述
3. `check_kindred_active(state: GameState, card: dict) -> bool` — 检查 card.race ∩ state.last_turn_races 或 card.spellSchool ∩ state.last_turn_schools 非空
4. `apply_kindred(state: GameState, card: dict) -> GameState` — 主入口，检测→检查条件→解析奖励→分发效果
5. 奖励效果复用 battlecry_dispatcher 的效果协议（damage, buff, draw, summon 等）
6. 特殊处理 `kindred_double_next` 标志：蛮鱼挑战者设置此标志，延系触发时检查并执行两次

**效果解析参考** (来自29张卡的实际文本):
- "使你的其他随从获得突袭" → `buff:other_friendly:0:0:RUSH`
- "召唤一个本随从的复制" → `summon:copy:self`
- "法力值消耗减少（2）点" → `cost_reduce:next:2`
- "重复一次" → `repeat:1` (重新释放本卡效果)
- "使其获得法术伤害+2" → `buff:target:SPELL_DAMAGE:2`

**graceful degradation**: 所有 dispatch 点 try/except，失败跳过延系奖励，搜索继续

**依赖**: MT-1.1 (GameState 需要 corpses 和 kindred_double_next 字段)

### MT-1.3: corpse.py — 残骸资源系统

**目标**: 实现 DK 残骸资源的获取、消耗、效果解析

**新建文件**: `hs_analysis/search/corpse.py`

**关键实现点**:
1. `CorpseEffect` dataclass: `cost: int`, `is_optional: bool`, `effect_text: str`
2. `parse_corpse_effects(card_text: str) -> list[CorpseEffect]` — 解析"消耗N份残骸"模式
   - 正则: `消耗(\d+)份残骸` 或 `消耗最多(\d+)份残骸`
   - 判断是否可选：如果文本结构是"消耗X份残骸，[额外效果]"，通常可选
3. `parse_corpse_gain(card_text: str) -> int` — 解析"获得一份残骸" / "获得N份残骸"
4. `can_afford_corpses(state: GameState, cost: int) -> bool`
5. `spend_corpses(state: GameState, cost: int) -> GameState`
6. `gain_corpses(state: GameState, amount: int) -> GameState`
7. `has_double_corpse_gen(state: GameState) -> bool` — 检测法瑞克是否在场
8. `resolve_corpse_effects(state: GameState, card: dict) -> GameState` — 主入口，解析→检查→消耗→应用效果

**残骸获取时机** (在 rhea_engine death flow 中):
- 友方随从死亡时: `gain_corpses(state, 1)` 或 `gain_corpses(state, 2)` (法瑞克在场)

**残骸消耗模式** (来自23张DK卡的实际文本):
- "消耗2份残骸，使其获得黑暗之赐" → cost=2, 可选=False
- "消耗2份残骸，再获得+1/+1" → cost=2, 可选=True (不消耗也有效果，只是少+1/+1)
- "消耗最多8份残骸" → cost=1-8, 动态消耗

**graceful degradation**: 解析失败时 cost=0，不阻止出牌

**依赖**: MT-1.1 (GameState 需要 corpses 字段)

### MT-1.4: rhea_engine.py 集成 — kindred + corpse

**目标**: 将延系和残骸系统接入 RHEA 引擎

**修改文件**: `hs_analysis/search/rhea_engine.py`

**关键实现点**:
1. PLAY MINION 流程中，colossal 之后、battlecry 之前插入延系检测:
   ```
   insert minion → colossal → **kindred** → battlecry → herald → trigger → aura → imbue → outcast → quest
   ```
2. PLAY SPELL 流程中，spell resolution 前插入延系检测
3. death flow 中，resolve_deaths 之后添加残骸获取:
   ```
   resolve_deaths() → **gain_corpses(1 或 2)** → cascade check
   ```
4. _apply_card_effects() 中添加残骸效果解析（对DK卡牌）
5. 所有新集成点用 try/except 包裹

**依赖**: MT-1.2 (kindred.py) + MT-1.3 (corpse.py)

### MT-1.5: test_kindred.py — 延系测试

**目标**: 延系模块的单元测试 + 集成测试

**测试用例 (~15)**:

3. check_kindred_active — 种族匹配、法术派系匹配、无匹配
4. apply_kindred — 完整流程：条件满足→效果应用

6. integration — 与现有 GameState 的 last_turn_races/schools 配合

### MT-1.6: test_corpse.py — 残骸测试

**目标**: 残骸模块的单元测试 + 集成测试

**新建文件**: `hs_analysis/search/test_corpse.py`

## Batch 2: Rune + Dark Gift（依赖 Batch 1 的 corpse 系统）

### MT-2.1: rune.py — DK 符文系统

**目标**: 实现符文类型识别 + 发现池过滤 + 条件效果触发

**新建文件**: `hs_analysis/search/rune.py`

**关键实现点**:
1. `RUNE_MAP: dict[str, str]` — spellSchool → 符文名映射:
   - `FROST` → `冰霜符文`
   - `SHADOW` → `邪恶符文`
   - `FIRE` → `鲜血符文`
2. `RUNE_LOOKUP: dict[int, str]` — 硬编码已知符文卡牌的 dbfId → 符文名 (约5张卡)
3. `get_rune_type(card: dict) -> str | None` — 从 spellSchool 或 RUNE_LOOKUP 获取符文类型
4. `filter_by_rune(pool: list[dict], rune_name: str) -> list[dict]` — 过滤发现池
5. `check_last_played_rune(state: GameState, rune_name: str) -> bool` — 检查上张牌的符文类型
6. `resolve_rune_discover(state: GameState, card: dict) -> GameState` — 处理"发现一张XX符文牌"效果

**GameState 修改**: 添加 `last_played_card: dict | None = None` 到 GameState（在 rhea_engine PLAY action 中更新）

**注意**: 大部分 DK 卡没有可识别的符文类型 — 这没关系，符文主要用于发现过滤

### MT-2.2: dark_gift.py — 黑暗之赐系统

**目标**: 实现黑暗之赐附魔池 + 应用 + 手牌检测

**新建文件**: `hs_analysis/search/dark_gift.py`

**关键实现点**:
1. `DARK_GIFT_ENCHANTMENTS: list[dict]` — ~10 个预定义附魔:
   - 属性增益: +2/+2, +1/+3, +3/+1
   - 关键词: 风怒, 吸血, 圣盾, 嘲讽
   - 效果: "亡语：对一个随机敌人造成2点伤害", "战吼：抽一张牌"
2. `apply_dark_gift(card: dict) -> dict` — 随机选择附魔，返回增强后的卡牌（修改 attack/health 或添加 enchantment）
3. `has_dark_gift_in_hand(hand: list[dict]) -> bool` — 检测手牌中是否有黑暗之赐卡牌
4. `filter_dark_gift_pool(pool: list[dict], constraint: str) -> list[dict]` — 过滤可具有黑暗之赐的卡牌

**与 discover.py 集成** (在 MT-2.3 中实现):
- 当卡牌文本说"发现一张具有黑暗之赐的XX牌"时:
  1. 用 constraint 过滤池（亡语、龙、等）
  2. 对每个候选项调用 `apply_dark_gift(card)`
  3. 在增强后的卡牌中做 best-of-3 选择

**与 battlecry_dispatcher.py 集成** (在 MT-2.3 中实现):
- 当卡牌文本说"如果手牌中有具有黑暗之赐的随从牌"时:
  1. 调用 `has_dark_gift_in_hand(state.hand)`
  2. 如果 True，应用条件奖励

### MT-2.3: discover.py + battlecry_dispatcher.py 集成

**目标**: 将符文过滤和黑暗之赐集成到发现和战吼系统

**修改文件**:

3. 保持现有逻辑不变，仅添加新分支

## Batch 3: Target Selection Redesign（依赖 Batch 1+2 稳定）

### MT-3.1: battlecry_dispatcher.py — 穷举目标选择

**目标**: 替换贪心目标选择为穷举评估

**修改文件**: `hs_analysis/search/battlecry_dispatcher.py`

**关键实现点**:
1. 新增 `select_best_target(state, card, valid_targets, effect, evaluate_fn) -> target | None`:
   - 遍历所有 valid_targets (≤7)
   - 对每个 target: deepcopy(state) → apply_effect(copy, effect, target) → evaluate_fn(copy)
   - 选得分最高的 target
   - 无有效目标返回 None
2. `evaluate_fn` 参数: 从 rhea_engine 导入 `_evaluate_state` (或暴露为 public `evaluate_state`)
3. 保留原有 greedy 逻辑作为 fallback（穷举失败时回退）
4. 随机效果（random_enemy, random_minion）不变 — 无法枚举随机结果
5. 确定1效果（如 "对所有XX"）不变 — 无需选择

**性能**: max 7 次评估 × O(1) 评估函数 = 可忽略

### MT-3.2: spell_simulator.py — 法术目标选择

**目标**: 对法术的单体效果也使用穷举评估

**修改文件**: `hs_analysis/search/utils/spell_simulator.py`

**关键实现点**:
1. 同 MT-3.1 的模式: 遍历有效目标 → copy → evaluate → 选最优
2. 适用场景: 单体伤害、单体变形、单体buff
3. 不适用: AoE、随机目标、全体效果

### MT-3.3: test_target_selection.py — 目标选择测试

**新建文件**: `hs_analysis/search/test_target_selection.py`

**测试用例 (~10)**:
1. 单目标穷举 — 选择最优（如打脸致死 vs 打随从）
2. 多目标穷举 — 3-7 个有效目标时选最优
3. 无有效目标 — 返回 None
4. 随机效果 — 仍用原有逻辑
5. AoE 效果 — 不触发目标选择
6. fallback — 穷举失败时回退到贪心
7. 性能 — 7 目标场景 <1ms
